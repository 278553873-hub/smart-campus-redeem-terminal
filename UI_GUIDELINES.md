# 智能终端及移动端交互设计规范

本规范适用于智能终端、教师手机端、家长手机端等触屏界面。它不是不可挑战的绝对审美标准，而是项目经验沉淀。

后续涉及 UI（用户界面）设计、交互调整、字号、图标、间距、文案、布局时，应先使用专业 UI skill 做独立评审，再对照本文档判断哪些规则应保留、修正或降级。

---

## 1. 使用优先级

### 1.1 专业评审优先
- 修改界面前，优先使用已安装的 `frontend-design-review` 做设计评审。
- 评审重点包括：任务路径、视觉层级、字号体系、图标体系、触控热区、组件状态、一致性、无障碍。
- 如果专业评审意见与本文档冲突，不应直接以本文档为准；应先说明冲突点、影响范围和推荐取舍。

### 1.2 场景约束第二
- 必须结合真实使用场景判断：教师手机端、家长手机端、智能终端、货柜机横屏小屏等场景的优先级不同。
- 移动端优先考虑触控效率、识别速度和误触风险。
- 终端机优先考虑商品/兑换内容的屏占比、滑动效率和大面积触控。

### 1.3 本文档作为项目基线
- 本文档用于记录项目已验证的经验和约束。
- “硬性约束”默认执行，除非专业评审或实际验证证明它不适合当前场景。
- “设计建议”允许被专业 skill、截图评审、用户反馈和实际验证挑战。

---

## 2. 硬性约束

硬性约束与设备、平台、可用性、误触风险相关。改动前应优先遵守；如需偏离，必须说明原因。

### 2.1 触屏优先
- 智能终端和手机端默认运行在触屏设备上，交互反馈应符合真实触屏体验。
- 所有关键可点击元素必须有明确点击反馈，避免用户不知道是否已触发操作。
- 反馈应快速、克制，建议动画时长不超过 150ms。

### 2.2 禁止依赖鼠标悬停
- 触屏端不得把核心交互建立在 `hover:`、`group-hover:` 等鼠标悬停状态上。
- 如果历史代码中存在 `hover:`，需要判断它是否影响触屏体验；对核心操作、列表项、按钮，应改为 `active:` 或明确的选中态。
- PC（个人电脑）后台和大屏不适用本条，应参考 `PC_UI_GUIDELINES.md`。

### 2.3 必须提供按压反馈
- 按钮、卡片、列表项等可交互元素应提供 `active:` 按压反馈。
- 常用反馈包括：`active:scale-95`、`active:opacity-70`、`active:bg-slate-100`。
- 同一页面内同类组件的按压反馈应统一，避免不同入口像来自不同系统。

### 2.4 触控热区
- 手机端和终端端的关键点击区域建议不小于 44px * 44px。
- 纯图标按钮不能只让图标本身可点，应给外层容器足够热区。
- 相邻点击区之间应保留足够间距，避免误触。

### 2.5 返回与关闭路径
- 独立页面应提供明确返回路径。
- 弹窗、抽屉、浮层必须提供清晰关闭方式。
- 关键流程中不得出现无法返回、无法取消、无法关闭的死路。

### 2.6 浮层遮挡与可读性
- 悬浮栏、顶部栏、底部操作条、弹窗遮罩等组件必须确保底层文字不会透出造成阅读干扰。
- 如使用毛玻璃效果，应验证滚动时底层图文不会与前景文字黏连。
- 多层叠加时，应通过边界、阴影、遮罩或透明度建立清晰层级。

### 2.7 小程序头部基础约束
- 手机端目前语义上模拟微信小程序，而不是独立 App。
- 如果页面使用项目统一小程序壳子，应避免在页面内部重复添加状态栏安全区。
- 如需复用微信右侧胶囊按钮，应保证标题、返回按钮、胶囊按钮的布局不冲突。

## 3. 设计建议

设计建议与审美、风格和历史经验相关。它们可以作为默认方向，但允许被专业评审、截图验证和用户反馈调整。

### 3.1 字号体系
- 同一页面应控制字号层级数量，避免出现过多相近字号。
- 建议先定义页面标题、分组标题、正文、辅助说明、数字/金额、标签等角色，再分配字号。
- 同一角色的字号、字重、行高应统一。
- 如果专业评审认为当前字号层级不清晰，应优先输出层级表，再修改代码。

### 3.2 图标体系
- 同级入口的图标尺寸、线宽、容器尺寸、圆角和颜色强度应统一。
- 图标与文字的间距应形成规律，避免局部松散或拥挤。
- 图标只作为辅助识别时，应避免过度抢占文字主信息。
- 涉及“我的页面”等入口型页面时，应重点检查图标尺寸是否成组一致。

### 3.3 内容一致性与整体感
- 相关状态、说明和操作应内聚在同一个视觉容器中，避免生硬拼接。
- 不同模式或状态下的操作栏需要具备可识别的视觉差异。
- 如果一个模块看起来像另一个产品或另一个设计系统，应优先统一组件结构和视觉 token（设计变量）。

### 3.4 毛玻璃与视觉层级
- 毛玻璃可用于顶部栏、底部操作条、弹窗等浮动组件，但不应机械套用到所有场景。
- 是否使用高强度毛玻璃，应由可读性、层级遮挡和整体风格决定。
- 如果毛玻璃削弱性能、可读性或造成视觉噪声，应改用更清晰的白底、半透明底或实色遮罩。

### 3.5 卡片、间距与视觉节奏
- 卡片内边距、列表项高度、分组间距应成体系，不应凭感觉混用。
- 手机端应优先保证信息读取效率，避免为了“呼吸感”牺牲核心内容展示。
- 同一页面内相同类型卡片的圆角、阴影、边框和背景应一致。

### 3.6 货柜机终端空间利用
- 横屏或小屏货柜机端空间有限，应优先让位于核心商品内容。
- 如果某个分类标题或说明已不再提供有效信息，应删除，提升商品展示面积。
- 终端端的滚动和点击路径应尽量短，减少用户在机器前停留和误触。

## 4. UI 评审清单

在生成代码或修改组件之前，先完成以下检查：

1. 是否已使用 `frontend-design-review` 做独立评审？
2. 是否说明了专业评审与本文档的冲突点？
3. 字号角色是否清晰，是否存在同级字号混用？
4. 同级图标的尺寸、容器、颜色、线宽是否统一？
5. 是否避免依赖 `hover:` 完成触屏端核心交互？
6. 是否为关键交互提供 `active:` 按压反馈？
7. 触控热区是否接近或达到 44px * 44px？
8. 弹窗、抽屉、浮层是否有明确关闭路径？
9. 浮层背景是否保证底层文字不会干扰阅读？
10. 如果偏离本文档，是否记录了原因和验证方式？

---

## 5. 规范更新原则

- 如果专业 skill 多次指出本文档某条建议不合理，应更新本文档。
- 如果某条规则只适用于终端机，不应强制套用到手机端。
- 如果某条规则只适用于手机端，不应强制套用到 PC 后台或大屏。
- 修改规范时，应把结论记录到 `logs/agent_detail`，便于后续追溯。
