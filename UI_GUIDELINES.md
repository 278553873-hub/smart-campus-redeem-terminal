# 智能终端及移动端交互设计规范 (UI/UX Guidelines)

## 🎯 核心交互原则：触屏优先 (Touch First)

本项目（含终端机界面与移动端 App 面板）完全运行在**触控设备**上。所有的 UI 交互与视觉反馈必须遵循真实的触屏体验。

### 🚫 禁止使用鼠标悬停 (Hover)
- **绝对禁止**在任何组件或系统中使用 `hover:` 相关的 CSS 类（如 `hover:bg-blue-500`、`hover:text-white`、`group-hover:` 等）。
- 触屏设备不存在鼠标悬停的物理动作，使用 hover 类不仅无用，还会导致长按或滑动时出现不可预期的“粘滞”状态。

### ✅ 强制使用点击/按压反馈 (Active)
- 所有可交互元素（按钮、卡片、列表项等）**必须**提供即时的触感反馈。
- 统一使用 `active:` 修饰符来传达按压状态（例如：`active:scale-95`、`active:opacity-70`、`active:bg-slate-100` 等）。
- 确保反馈动画迅速且不拖沓（建议过渡时间 `< 150ms`）。

## 🎨 视觉与空间布局 (Visual & Layout)

### 1. 毛玻璃与层级遮挡 (Glassmorphism & Z-Index)
- **浮动/悬浮组件**（如底部的 Input Bar、顶部的 Header、弹出的 Modal）**必须**具备高强度的毛玻璃背景（例如 `backdrop-blur-xl`、`bg-white/90`），确保底部滚动的列表文字、图片绝对不可透射造成视线干扰。
- 如果出现多层叠加，必须具有明显的边界对比（如使用 `border` 或微妙的内部 `shadow-inner` / 外部投影）。

### 2. 交互元素的边界与热区 (Touch Targets)
- 任何按钮/可点击区域的大小不得小于 `44px * 44px`（遵循 Apple 规范）。
- 各个可点击区块之间需要有足够的间距，不要拥挤叠放，避免误触。
- 不要在内容滑动的直接路径上放置没有明确边界或太容易被“滑”到的悬浮按钮，以免误触发出。

### 3. 内容一致性与整体感 (Cohesive Unit)
- **不要为了拆分功能而做生硬的拼接**。相关的状态标签和操作按钮应当内聚在同一个视觉容器（Card 或 Panel）中，形成一个坚实的整体。
- 不同模式或状态下的操作栏需要具备明显的视觉反差（如颜色或主从色调），让用户一瞥即知当前所处的上下文。

### 4. 小程序原生规范限制 (Mini-Program Native Standards)
- **全局环境标识**：目前移动端包含“教师手机端”和“家长手机端”，环境在语义上模拟的是“微信小程序”而不是独立 App。
- **统一的头部高度 (Header)**：
  - 手机端（教师/家长）的 Header 必须严格保持 `44px` 高度（CSS 类名 `h-[44px]`，配合内边距 `px-4 py-2`）。
  - **切勿**在 Header 的容器上添加额外的顶级向下的安全边距（如 `pt-12`），因为全局壳子（App Mockup）已经统一预留了状态栏的安全区。
- **强制右侧胶囊按钮 (WeChat Capsule)**：
  - 小程序的每个独立页面顶栏**必须**包括原生的右侧安全胶囊按钮（包含 “...” 更多按钮和 “O” 关闭按钮），占用固定的宽度与位置。
  - 标准布局必须是：左侧（返回按钮） —— 中间（标题居中） —— 右侧（胶囊按钮组靠右对齐）。如果不遵循此要求会导致布局在原生微信不一致。

### 5. 货柜机终端空间利用 (Vending Machine Space Limitations)
- **极限空间显示**：横屏/小屏显示的货柜机端物理空间极为有限，必须让位于核心商品内容。
- **信息展示密度与去重**：如果某个分类维度（如不再区分特殊商品后，只卖货柜商品），它的说明就变成了冗余信息，此时**必须全面移除累赘的标题栏（诸如“全部货柜商品”文本框）**，从而最大化主要商品的屏占比大小和滚动的体验感。

## 🛠 开发自检清单 (Pre-Flight Checklist)
在生成代码或修改组件之前，请务必核对：
1. [ ] 是否移除了所有的 `hover:` 类名？
2. [ ] 是否添加了 `active:` 以提供按压反馈？
3. [ ] 悬浮栏/模态框后方的图文在滚动时是否会被清晰遮盖，不会造成“文字黏连”？
4. [ ] 所有的可交互入口（输入框、语音、拍照）是否在语义和样式上形成了完整的整体？
