# Agent 工作规范

## 交流语言
- 始终使用中文与用户交流与输出结果。

## 解决问题流程
- 先分析问题，明确目标与影响范围。
- 尽可能复现问题路径（含环境、步骤、输入、期望与实际结果）。
- 复现后再探讨问题原因与可选方案，应优先讨论。
- 讨论完后再确认修改代码与验证结果。
- 最后询问是否提交代码。

## 设计与规范
- 涉及界面、交互、视觉层级、信息密度、文案删减、布局调整等设计类任务时，必须先读取对应的 UI 规范 md，再开始设计或改代码。
- PC 后台与大屏相关设计，优先阅读 `PC_UI_GUIDELINES.md`。
- 如果存在多份规范，必须选择与当前模块最匹配的那一份，禁止凭感觉直接改界面。
- 修改结果如果与 UI 规范冲突，应以 UI 规范为准；如果规范本身不够明确，先补充规范再继续改动。

## 沟通日志与记忆增强
- 应该保持时常记录沟通日志的习惯。
- 在交流过程中，记录任何与问题相关的关键信息为日志，用于后续快速定位与减少重复分析，保证你的日志对日后工作是有用的，能提升工作效率的，注意细节。
- 日志保存位置：`/Users/mayday/project/校园智能积分兑换终端/logs/agent_detail`，永远不要提交这个日志的内容。
- 后续遇到相关问题时，优先读取并引用对应日志，而不是从头分析。
- 相似的内容尽量存放在一起，记录好更新信息，查阅。

## git规范
- commit应始终保持中文描述。

## 注意
- 每次修改配置后，注意同步.env.example和.env文件，清理没用的配置，添加新增的配置，注意增加注释说明。
- 日志是有用的有细节的，用户的重复问题不用从头分析
