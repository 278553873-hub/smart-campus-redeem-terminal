# Agent 工作规范

## 交流语言
- 始终使用中文与用户交流与输出结果。
- 你的服务对象是产品经理，而非技术人员，因此如用英文简写需要通过括号说明该简写的含义，比如DOM,CLS这些都应该增加说明

## 解决问题流程
- 先分析问题，明确目标与影响范围。
- 尽可能复现问题路径（含环境、步骤、输入、期望与实际结果）。
- 复现后再探讨问题原因与可选方案，应优先讨论。
- 讨论完后再确认修改代码与验证结果。
- 最后询问是否提交代码。

## 代码架构规范
- 设计必须遵循高内聚、低耦合原则，严格做好分层架构与职责拆分。
- 严格区分底层基础层、通用组件层、业务定制层，禁止业务代码直接侵入基础组件；
- 通过标准化架构与分层设计，从根源解决同类型问题重复出现，保障后续迭代可维护、可扩展。

## 设计与规范
- 涉及界面、交互、视觉层级、信息密度、文案删减、布局调整等设计类任务时，必须先明确目标页面、设备场景和影响范围，再开始设计或改代码。
- 涉及选择或调用全局 skill 时，先查看 `/Users/mayday/.codex/skills/INSTALLED_SKILLS_INDEX.md`，按其中的来源、用途和使用场景路由选择合适 skill，避免同时调用多个理念冲突的 skill。
- 遇到多种 UI/产品 skill 都可能适用时，先判断任务场景，再选择一个主 skill 和必要的辅助 skill；例如 iOS 高定风格优先 Apple HIG + `ui-ux-pro-max`，B 端/SaaS 后台优先 PM skills + `shadcn`，高视觉官网/落地页优先 `ui-ux-pro-max` + ibelick UI skills。
- 已安装专业 UI skill 时，应先使用专业 UI skill 做独立评审，再对照项目 UI 规范判断是否采纳；项目规范不是不可挑战的绝对审美标准。
- 手机端、智能终端相关设计，优先使用 `frontend-design-review` 审视字号体系、图标体系、触控热区、视觉层级、组件状态和无障碍，再读取并对照 `UI_GUIDELINES.md`。
- PC 后台与大屏相关设计，优先阅读 `PC_UI_GUIDELINES.md`。
- 如果存在多份规范，必须选择与当前模块最匹配的那一份，禁止凭感觉直接改界面。
- 修改建议如果与 UI 规范冲突，必须先说明冲突点、影响范围、专业评审依据和推荐取舍；不要直接机械套用旧规范。
- 如果规范本身不够明确，先补充或修正规范再继续改动。
- 对字号、图标、间距、颜色、圆角、阴影等一致性问题，应先输出评审结论与修改建议，经用户确认后再改代码。
- 全局弹窗、浮层类组件统一抽象封装，收敛通用样式与基础能力，杜绝局部临时写死、碎片化样式实现

## 沟通日志与记忆增强
- 应该保持时常记录沟通日志的习惯。
- 在交流过程中，记录任何与问题相关的关键信息为日志，用于后续快速定位与减少重复分析，保证你的日志对日后工作是有用的，能提升工作效率的，注意细节。
- 日志保存位置：`/Users/mayday/project/校园智能积分兑换终端/logs/agent_detail`，永远不要提交这个日志的内容。
- 后续遇到相关问题时，优先读取并引用对应日志，而不是从头分析。
- 相似的内容尽量存放在一起，记录好更新信息，查阅。

- 日志是有用的有细节的，用户的重复问题不用从头分析
