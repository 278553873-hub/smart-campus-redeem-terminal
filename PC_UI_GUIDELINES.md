# PC 后台 UI 规范

适用范围：使用 Arco Design React 作为基础组件库的 PC 后台、管理端、数据配置类页面。  
组件基准：`@arco-design/web-react` 与 `@arco-design/web-react/dist/css/arco.css`。  
使用对象：AI 编码助手、产品经理、前端开发。  
核心目标：在需求渐进明确的 vibe coding 场景中，让 AI 能基于统一规则做出稳定、克制、可维护的 UI 决策。

---

## 0. 使用方式

### 0.1 用户要求优先
- 当用户已明确字段、流程、交互或视觉要求时，优先执行用户要求。
- 当用户未明确时，AI 应基于本规范选择合理默认方案，不应因缺少完整 PRD 而停止工作。PRD 指产品需求文档，用来描述页面目标、字段、流程和业务规则。
- 用户可以用渐进式方式描述需求；AI 应支持“先做一版、再看、再调”的工作方式。

### 0.2 需求不完整时的处理
AI 应按风险分级处理缺失信息：
- **低风险**：布局、间距、对齐、组件替换、视觉层级、文案精简，可直接按规范处理。
- **中风险**：影响用户操作路径、字段展示、筛选逻辑，应说明默认假设后继续。
- **高风险**：影响数据计算、权限、资金/积分、不可逆操作、核心业务规则，必须先澄清。

### 0.3 Arco 使用边界
- 优先使用 Arco 组件和默认样式，不主动重写基础组件。
- AI 可以根据页面需要调整布局、宽度、间距和组件组合方式。
- 当确实需要覆盖 Arco 组件内部样式时，AI 应在回复中说明影响范围和原因，并优先采用局部覆盖，避免全局污染。
- 用户不需要为每个视觉调整解释原因；AI 负责判断该调整属于布局优化、组件配置，还是组件样式覆盖。

### 0.4 本规范不重复 Arco 文档
- Arco 已提供的基础能力不在本规范中重复解释。
- 本规范只记录本类后台页面的使用约定，例如组件选择、回显策略、筛选区组织、操作层级和信息密度。
- 查组件 API 时以 Arco 官方文档为准：
  - Select：https://arco.design/react/components/select
  - Cascader：https://arco.design/react/components/cascader
  - Table：https://arco.design/react/components/table
  - Form：https://arco.design/react/components/form
  - Button：https://arco.design/react/components/button
  - Modal：https://arco.design/react/components/modal
  - Drawer：https://arco.design/react/components/drawer

## 1. 设计原则

### 1.1 信息优先
- PC 后台页面优先服务判断、筛选、配置、录入、查看和操作结果。
- 不展示不能帮助用户判断、操作、纠错或理解风险的信息。
- 每新增一段标题、说明、卡片或装饰，都必须能回答：“用户看了以后能做出什么更准确的判断或动作？”

### 1.2 专业克制
- 后台界面以清晰、稳定、效率为核心，不追求营销化装饰。
- 避免非功能性渐变、大投影、大图标头图、彩色装饰块。
- 视觉层级应来自结构、间距、字重和状态，而不是堆叠装饰。

### 1.3 对齐一致
- 对齐是后台页面的基础。卡片边界、筛选项、表单标签、表格列边沿应保持清晰对齐关系。
- 表格表头与表体单元格的对齐方式必须一致。
- 同一视觉组内的同类控件应保持一致宽度策略，但不为了对齐强行拉长短字段。

### 1.4 文案克制
- 后台页面默认不写描述性副标题。
- 字段名、区块名、表头已经能表达含义时，不再补充说明。
- 禁止使用“用于管理……”“可进行……配置”“支持……操作”等低信息量句子。
- 只有存在风险、限制、数据口径、权限差异、不可逆影响、异常状态时，才使用 Alert 或辅助说明。

## 2. 页面结构

### 2.1 后台壳层
- 是否使用侧边栏、顶部栏、双栏布局，由项目导航复杂度决定。
- 多模块后台优先使用侧边栏 + 顶部栏。
- 单模块工具型后台可使用顶部导航或轻量侧边栏。
- 壳层尺寸一旦确定应全站一致，不在业务页面中单独修改基础导航尺寸。

### 2.2 列表页
列表页默认结构：
1. 页面定位信息，如面包屑。
2. 主卡片标题。
3. 筛选区。
4. 分割线。
5. 主操作区。
6. 表格。
7. 分页。

使用规则：
- 筛选区只放高频必要条件；低频条件应考虑折叠到“更多筛选”。
- 主操作区通常放新增、批量导入、批量导出等页面级操作。
- 表格行内操作只放当前行相关操作。

### 2.3 表单页
- 强流程、强顺序、字段之间有依赖关系时，优先单列布局。
- 字段独立、数量较多、用户需要快速扫视时，可使用两列布局。
- 长文本、上传、复杂选择器、表格型录入控件应独占整行。
- 关键字段不应为了凑两列而被拆散。

### 2.4 详情页
- 查看页优先呈现结果，不暴露编辑态干扰。
- 局部导出、局部操作应放在对应分组标题右侧，作用范围必须清晰。
- 如果详情内容需要频繁对照列表，可使用 Drawer；如果内容复杂、需要独立 URL 或多步骤，可使用独立页面。

### 2.5 配置页
- 配置页应减少大段说明，把空间留给配置项、状态和保存结果。
- 多条同类配置优先表格化，不用重复卡片堆叠。
- 配置保存后影响范围较大时，必须展示清晰提示。

## 3. 布局与密度

### 3.1 间距体系
- 页面、卡片、表单、列表统一使用 8px 间距体系。
- 允许 4px 作为局部微调。
- 避免 10px、15px、19px 等随机值。
- 卡片之间、筛选区与表格区之间，应通过稳定间距和分割线形成层级。

### 3.2 卡片使用
- 列表页、配置页、管理页默认采用一个主白色卡片承载核心内容。
- 不为了“分组感”拆出多个大卡片；只有当内容属于不同任务、不同流程或不同权限边界时，才允许拆分。
- 主卡片与表格容器保持轻量边界，不使用强投影和大圆角装饰。

### 3.3 控件宽度决策
控件宽度由信息长度、输入复杂度、使用场景决定，不由字段名称决定。

- 筛选栏短枚举：160px - 220px。
- 筛选栏多选对象：200px - 280px，避免过宽挤占查询按钮。
- 表单标准输入：320px - 420px。
- 长文本、上传、表格、复杂选择器：独占整行或使用 18/24 - 24/24 栅格。
- 进度条、Alert、横向说明等中宽内容不应在宽屏下无限拉长。

### 3.4 筛选区组织
- 筛选项由业务需求决定，本规范不规定某个页面必须有哪些筛选字段。
- 默认排序建议：时间/范围 -> 类型/状态 -> 组织对象 -> 内容对象 -> 创建人/操作人 -> 查询/重置。
- 查询按钮使用主按钮；重置按钮使用默认按钮。
- 多选筛选应支持清空；选项较多时支持搜索。
- 筛选栏多选回显应保持单行稳定，避免大量 Tag 撑高筛选区。

### 3.5 操作区组织
- 一个操作区只保留 1 个视觉主按钮。
- 页面级操作放在表格上方或卡片标题右侧。
- 行内操作只影响当前行；分组标题右侧操作只影响当前分组。
- 危险操作不应和主操作同等强调，应使用二次确认或收进更多菜单。

## 4. Arco 组件使用约定

### 4.1 Button 按钮
- 使用 Arco Button，不手写基础按钮样式。
- 页面主操作使用 `type="primary"`。
- 次要操作使用默认 Button。
- 危险操作使用 `status="danger"` 或二次确认。
- 表格行内操作优先使用文字按钮；超过 2-3 个操作时考虑“更多”菜单。

### 4.2 Select 选择器
适用场景：
- 平级枚举选择。
- 类型、状态、科目、标签等没有父子层级的数据。

使用约定：
- 单选使用 Arco `Select`。
- 多选使用 `mode="multiple"`。
- 可清空场景开启 `allowClear`。
- 选项较多时开启 `showSearch`。
- 筛选栏多选使用 `maxTagCount` 或数量汇总，避免撑高筛选栏。
- 表单确认场景如需用户核对，可展示更多 Tag。
- 不把占位文案作为真实 Option。

### 4.3 Cascader 级联选择
适用场景：
- 有明确父子层级的数据。
- 例如组织 -> 部门、年级 -> 班级、地区 -> 学校。

使用约定：
- 使用 Arco `Cascader`。
- 多选使用 `mode="multiple"`。
- 如果业务只需要叶子节点，使用 `checkedStrategy="child"`。
- 可清空场景开启 `allowClear`。
- 选项较多时开启 `showSearch`。
- 筛选栏中不展示过多 Tag，使用 `maxTagCount` 或数量汇总。
- 列宽通过 `dropdownMenuColumnStyle` 控制，不手写浮层结构。

不适用场景：
- 平级枚举不要用 Cascader。
- 无稳定层级关系的数据不要强行做级联。

### 4.4 Table 表格
- 使用 Arco Table；特殊可编辑表格、Excel 粘贴表格等业务复杂场景可自定义。
- 表头与表体对齐方式必须一致。
- 文本列默认左对齐；用于大小比较的数值列可右对齐。
- 表格只展示用户判断和操作所需字段，不把所有数据库字段直接铺出。
- 操作列保持稳定宽度，避免因按钮数量变化导致表格抖动。
- 空状态、加载状态、错误状态必须明确。

### 4.5 Form 表单
- 使用 Arco Form 组织标签、校验、错误提示和布局。
- 必填项、校验错误、禁用原因必须清晰可见。
- 强关联字段应相邻；有依赖关系的字段应按用户理解顺序排列。
- 不因视觉对齐强行打乱业务理解顺序。

### 4.6 Modal 与 Drawer
Modal 适用：
- 短流程。
- 强确认。
- 不需要频繁参考原页面上下文。
- 操作完成后回到原页面。

Drawer 适用：
- 需要保留列表上下文。
- 查看详情、快速编辑、侧边配置。
- 内容比 Modal 长，但不值得跳新页面。

独立页面适用：
- 多步骤流程。
- 复杂表单。
- 大量数据录入。
- 需要稳定 URL 或刷新恢复。

### 4.7 Alert 提示
- 使用 Arco Alert。
- 只有当提示能帮助用户避免错误、理解风险、掌握关键规则时使用。
- 不用 Alert 重复页面标题、字段名或一眼可见的操作说明。
- 风险、不可逆、权限限制、数据口径说明优先使用 Alert。
- 长期有效的规则提示不可关闭；临时提示可关闭。

### 4.8 Popconfirm、Message、Notification
- 行内危险操作优先使用 Popconfirm 二次确认。
- 即时成功或失败反馈使用 Message。
- 需要保留更长时间、包含标题和说明的反馈使用 Notification。
- 不用全局提示替代表单字段级错误。

## 5. 业务复合组件设计原则

### 5.1 自定义业务组件边界
- Arco 已提供的基础组件不重新手写。
- 只有业务复合组件才允许自定义，例如可编辑表格、批量录入表格、复杂配置画布。
- 自定义组件应优先由 Arco 基础组件组合而成。
- 如果自定义组件包含特殊规则，应写入页面设计说明，而不是放进通用 UI 规范。

### 5.2 可编辑表格
- 可编辑表格必须明确普通、悬浮、聚焦、选中、错误状态。
- 序号列、计算列、系统字段不可编辑。
- 批量操作必须明确作用范围，避免用户误解为全局操作。
- 支持粘贴、拖拽、批量填充等高级能力时，应提供轻量提示。
- 键盘操作规则应在页面设计说明中明确。

### 5.3 分组选择
- 候选项数量有限、用户需要同时看到全部候选项、且选择行为高频时，可使用分组多选面板。
- 优先确认 Arco Select / Cascader / TreeSelect 是否满足。
- 只有标准组件无法满足“同时浏览 + 快速批量选择”时，才自定义分组多选面板。
- 自定义面板应保持选中/未选中尺寸一致。
- 分组标题只做分类，不承载交互，除非需求明确需要组级全选。

### 5.4 Sheet / Tab 类结构
- Sheet / Tab 用于同一任务下的并列对象切换，不用于承载完全不同任务。
- 当前态必须明显，但不依赖“当前”等额外文案拉长标题。
- 切换动效不应带动下方内容区域抖动。
- 只影响当前 Sheet / Tab 的操作必须放在当前内容区内，避免误解为全局操作。

## 6. 文案与状态

### 6.1 按钮文案
- 按钮文案使用动词 + 对象，例如“导出 Excel”“填充学生姓名”。
- 避免“确定”“处理”“操作”等模糊文案，除非上下文已经非常明确。
- 危险操作文案应明确后果，例如“删除记录”“停用账号”。

### 6.2 说明文案
- 说明文案必须提供额外决策价值。
- 不复述标题，不解释一眼可见的功能。
- 涉及规则、限制、风险、影响范围、数据口径时才写说明。

### 6.3 空、加载、错误状态
- 空状态说明当前为什么为空，并给出下一步动作或判断。
- 加载状态应避免页面布局大幅跳动。
- 错误状态应说明原因和可执行修复动作。
- 禁用状态应让用户知道为什么不能操作。

## 7. 自检清单

AI 在交付 PC 后台页面或改动前，应自检：

- [ ] 是否优先使用 Arco 组件，而不是手写基础组件？
- [ ] 用户已明确的字段、流程、交互或视觉要求是否被优先执行？
- [ ] 需求缺失是否按风险分级处理，而不是机械要求补全 PRD？
- [ ] 当前组件选择是否符合数据结构：平级用 Select，层级用 Cascader / TreeSelect？
- [ ] 筛选区是否只展示高频必要条件，低频条件是否考虑折叠？
- [ ] 多选回显是否避免撑高筛选栏？
- [ ] 表格字段是否只展示用户判断和操作所需信息？
- [ ] 表头与表体对齐方式是否一致？
- [ ] 操作按钮是否有主次层级？
- [ ] 危险操作是否有二次确认？
- [ ] 说明文案是否提供真实决策价值？
- [ ] 页面是否包含加载、空、错误、禁用等基础状态？
- [ ] 如覆盖 Arco 组件内部样式，是否说明影响范围并采用局部覆盖？
