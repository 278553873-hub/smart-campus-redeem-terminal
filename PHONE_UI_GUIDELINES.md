# 手机端 UI/UX 设计系统规范

本规范仅适用于项目中的手机端 Demo 页面，包括教师手机端、家长手机端，以及以手机尺寸展示的小程序/H5 页面。不适用于智能终端、货柜机、大屏、PC 后台或其他非手机尺寸设备。

本规范目标不是让单个页面“看起来更好”，而是建立一套可复用、可维护、可扩展的手机端视觉与交互系统。后续手机端页面设计与改造，应优先遵循本规范；如规范无法覆盖真实场景，应先补充规范或新增语义组件，再修改页面。

---

## 1. 适用范围

### 1.1 适用

- 教师手机端 Demo。
- 家长手机端 Demo。
- 手机尺寸下的小程序模拟页面。
- 手机尺寸下的 H5 页面。
- 手机端页面中的弹窗、底部抽屉、顶部栏、底部操作区、菜单、卡片、列表、表单、报告预览等组件。

### 1.2 不适用

- 智能兑换终端。
- 货柜机横屏或竖屏终端。
- 教室大屏。
- PC 后台。
- SaaS 管理后台。
- 大屏数据驾驶舱。

这些设备的屏幕尺寸、观看距离、触控方式、信息密度和操作路径与手机差异较大，不应被本规范包裹。终端类界面应另行制定专门规范。

---

## 2. 核心原则

### 2.1 真实手机场景优先

手机端页面应优先满足真实使用场景：

- 教师在移动状态下快速记录学生表现。
- 教师查看班级、学生、记录、文件和报表。
- 家长查看学生成长信息。
- 用户在小程序壳层中进行返回、关闭、切换、提交等操作。

手机端设计必须优先保证：

- 信息识别快。
- 点击不误触。
- 层级稳定。
- 页面之间具有统一系统感。
- 同一组件跨页面复用时不需要重写样式。

### 2.2 三层架构

手机端 UI（用户界面）必须分为三层：

| 层级 | 职责 | 示例 | 禁止 |
|---|---|---|---|
| 基础视觉层 | 定义颜色、字体、圆角、阴影、间距、图标尺寸、动效 | `font.body`、`radius.card`、`shadow.card` | 禁止出现业务命名 |
| 通用组件层 | 封装按钮、卡片、菜单项、图标徽章、弹窗、顶部栏 | `MobileCard`、`IconBadge`、`MenuItem` | 禁止写死业务内容 |
| 业务组合层 | 根据具体页面组合通用组件 | 我的页面、记录页、报表页 | 禁止复制硬编码样式 |

### 2.3 高内聚、低耦合

- 基础 token（设计变量）只负责视觉值，不关心业务。
- 通用组件只负责结构、状态、交互和通用视觉。
- 业务页面只负责传入内容、选择组件变体、组织页面逻辑。
- 一个业务页面不得直接复制另一页面的大段样式。
- 同类组件必须通过 `variant`、`size`、`tone` 等有限参数变化，不得重新写一套 class。

### 2.4 组件先于页面

当多个页面出现相似结构时，应优先沉淀组件，而不是继续按页面复制：

- 卡片相似时，沉淀 `MobileCard`。
- 菜单分组相似时，沉淀 `MenuSection`。
- 菜单项相似时，沉淀 `MenuItem`。
- 图标容器相似时，沉淀 `IconBadge`。
- 底部抽屉相似时，沉淀 `BottomSheet`。

## 3. 字体系统

### 3.1 字体族

手机端默认字体：

```text
PingFang SC, Hiragino Sans GB, Microsoft YaHei, Noto Sans CJK SC, sans-serif
```

数字、编号、积分、金额可使用等宽或几何数字字体，但不得影响中文正文阅读。

### 3.2 字体角色

手机端常规页面只允许以下字体角色：

| token | 字号 | 字重 | 行高 | 使用场景 |
|---|---:|---:|---:|---|
| `text.navTitle` | 17px | 700 | 1.2 | 小程序顶部标题 |
| `text.pageTitle` | 24px | 800 | 1.15 | 页面主标题、个人姓名、关键页标题 |
| `text.sectionTitle` | 17px | 600/700 | 1.25 | 分组标题、卡片标题 |
| `text.itemTitle` | 15px | 400/500 | 1.35 | 列表主标题、菜单项标题 |
| `text.body` | 14px | 400/500 | 1.55 | 正文、说明、表单内容 |
| `text.helper` | 12px | 400/500 | 1.4 | 时间、备注、辅助说明 |
| `text.label` | 11px | 600/700 | 1.2 | 标签、状态、徽章 |
| `text.metric` | 28px/32px | 800/900 | 1.0 | 积分、金额、核心统计数字 |

### 3.3 字体使用规则

- 普通页面不得新增 `13px`、`16px`、`18px` 等临时字号。
- `font-black` 只允许用于页面主标题、核心数字、极少量强状态标签。
- 设置类、个人中心类菜单项默认使用 400 字重；只有内容列表主标题或需要强调的入口才使用 500 字重。
- 菜单项不得使用 `font-black`，应使用 `text.itemTitle`。
- 辅助说明不得低于 12px，除非是徽标、角标、报告页页码等极弱信息。
- 同一页面中常规字号不得超过 4 种。
- 同一角色跨页面必须保持字号、字重、行高一致。

### 3.4 字体例外

以下场景允许使用额外字号，但必须封装在对应组件内：

- 报告封面、证书、海报类手机预览页。
- 手机壳状态栏、微信胶囊模拟层。
- 空状态插画标题。
- AI 结果卡中的强调型评分数字。

例外字号不得直接散落在业务页面里。

## 4. 圆角系统

### 4.1 圆角 token

手机端只允许以下圆角角色：

| token | 值 | 使用场景 |
|---|---:|---|
| `radius.sm` | 8px | 小标签、小按钮、进度条、弱徽章 |
| `radius.md` | 12px | 输入框、普通按钮、小型图标容器 |
| `radius.lg` | 16px | 列表项、普通图标容器、普通卡片 |
| `radius.xl` | 24px | 主卡片、分组卡片、数据卡片 |
| `radius.sheet` | 32px | 底部抽屉、弹窗、大型 Hero 卡片 |
| `radius.full` | 999px | 头像、胶囊按钮、圆形按钮 |

### 4.2 圆角使用规则

- 组件尺寸越大，圆角可以越大；小组件不得使用大圆角。
- 同一组件的默认态、选中态、禁用态不得改变圆角。
- 圆角用于表达组件层级，不用于表达业务状态。
- 禁止页面中随意使用 `rounded-[20px]`、`rounded-[28px]`、`rounded-[32px]` 等任意值。
- 如果一个任意圆角在 3 个以上位置重复出现，必须升级为正式 token 或组件变体。

### 4.3 组件映射

| 组件 | 圆角 |
|---|---|
| 标签、徽章 | `radius.sm` |
| 输入框、普通按钮 | `radius.md` |
| 普通列表项 | `radius.lg` |
| 普通内容卡片 | `radius.lg` / `radius.xl` |
| 页面分组卡片 | `radius.xl` |
| 个人信息 Hero 卡 | `radius.sheet` |
| 底部抽屉 | `radius.sheet` |
| 头像、胶囊 | `radius.full` |

## 5. 阴影系统

### 5.1 阴影 token

手机端只允许以下阴影角色：

| token | 使用场景 | 视觉要求 |
|---|---|---|
| `shadow.none` | 扁平列表、弱分隔区域 | 无阴影 |
| `shadow.card` | 普通白卡片 | 轻、短、低透明 |
| `shadow.raised` | 强调卡片、选中项 | 中等层级，不厚重 |
| `shadow.floating` | 底部悬浮栏、悬浮按钮 | 明确悬浮，但不遮挡内容 |
| `shadow.modal` | 弹窗、底部抽屉 | 层级最高，需配合遮罩 |

### 5.2 阴影使用规则

- 普通卡片默认使用 `shadow.card` 或无阴影，不得使用 `shadow-2xl`。
- 同一组卡片必须使用同一阴影等级。
- 彩色阴影不是基础阴影 token，只允许用于主操作按钮、当前选中态、AI 分析结果卡、成功/警告/危险等强反馈状态。
- 彩色阴影必须轻，不得制造厚重发光感。
- 禁止在页面中随意新增一次性 `shadow-[...]`。
- 如果需要特殊阴影，必须封装为语义组件，例如 `CardAiResult`、`BottomInputBar`。

## 6. 图标系统

### 6.1 图标风格

手机端默认图标体系使用线性图标。

推荐：

- 统一使用 Lucide 风格线性图标。
- 默认线宽为 2。
- 小尺寸状态图标可使用 2.2。
- 装饰图标可使用 1.5，但不得用于核心操作。

禁止：

- 同一组入口混用线性图标、实心图标、插画图标。
- 图标颜色与业务语义无关地随意变化。
- 图标本身承担全部点击热区。

### 6.2 图标尺寸 token

| token | 图标尺寸 | 容器尺寸 | 使用场景 |
|---|---:|---:|---|
| `icon.sm` | 16px | 32px | 标签、状态、分组辅助图标 |
| `icon.md` | 20px | 40px / 44px | 顶部返回、更多、普通操作 |
| `icon.lg` | 24px | 44px / 48px | 菜单入口、功能入口 |
| `icon.xl` | 28px | 56px | 空状态、强调入口、装饰场景 |

### 6.3 图标使用规则

- 顶部返回、关闭、更多：使用 `icon.md`，点击热区不小于 44px。
- 我的页面菜单入口：使用 `icon.lg`。
- 分组标题图标：使用 `icon.sm` 或 `icon.md`。
- 状态标签图标：使用 `icon.sm`。
- 空状态、插画辅助图标：使用 `icon.xl`，不得承担核心操作。
- 图标容器必须通过 `IconBadge` 统一控制尺寸、圆角、颜色和线宽。

## 7. 间距系统

### 7.1 间距 token

手机端间距采用 4px 基准，但常规页面只使用以下语义间距：

| token | 值 | 使用场景 |
|---|---:|---|
| `space.xs` | 4px | 图标与文字的紧密间距 |
| `space.sm` | 8px | 标签之间、弱元素之间 |
| `space.md` | 12px | 列表内部、小组件内部 |
| `space.lg` | 16px | 页面默认边距、卡片内部 |
| `space.xl` | 20px | 主卡片内部、模块间距 |
| `space.2xl` | 24px | 页面大分组、Hero 区域 |

### 7.2 间距使用规则

- 手机端页面左右默认边距为 16px。
- 信息密度较低的个人中心页面可使用 20px，但不得全局扩大。
- 卡片内边距优先使用 16px 或 20px。
- 同一页面内连续卡片形成“模块栈”时，卡片之间使用 12px 或 16px，不使用 20px 以上间距。
- Hero 卡片与下方第一组内容之间使用 16px，除非需要明确分隔两类完全不同任务。
- 设置类、我的页面类分组卡片之间优先使用 12px，参考主流手机设置页的紧凑列表节奏。
- 模块间距优先使用 16px；20px 仅用于跨任务区块或视觉上需要明显停顿的位置。
- 大面积 24px 间距仅用于页面主视觉、报告卡片、底部抽屉。
- 禁止为了“呼吸感”使用过大的垂直留白导致首屏信息减少。

## 8. 色彩系统

### 8.1 基础色

| token | 用途 |
|---|---|
| `color.bg` | 页面背景 |
| `color.surface` | 卡片、弹窗、底部栏 |
| `color.surfaceSubtle` | 卡片内弱分区 |
| `color.border` | 轻边界 |
| `color.textPrimary` | 一级文字 |
| `color.textSecondary` | 二级文字 |
| `color.textMuted` | 辅助文字 |
| `color.textDisabled` | 禁用文字 |

### 8.2 语义色

| token | 用途 |
|---|---|
| `color.brand` | 主操作、选中态、关键链接 |
| `color.success` | 成功、正向反馈 |
| `color.warning` | 警告、待处理 |
| `color.danger` | 删除、错误、风险操作 |
| `color.ai` | AI 分析、智能生成 |
| `color.classMode` | 班级模式 |
| `color.studentMode` | 学生模式 |

### 8.3 色彩使用规则

- 普通文字不得使用纯黑，应使用深蓝灰或 slate 系列。
- 颜色必须服务状态和层级，不得单纯装饰。
- 同一业务状态跨页面颜色必须一致。
- AI 相关颜色可以更鲜明，但必须收敛在 AI 组件内。
- 页面不得随意新增十六进制色值；新增颜色必须先判断是否能复用现有语义色。

## 9. 通用组件规范

### 9.1 MobilePage

统一手机端页面容器。

职责：

- 页面背景。
- 滚动容器。
- 底部安全区。
- 是否使用小程序顶部壳层。
- 是否允许背景延展到手机状态栏。

禁止：

- 不允许写入具体业务内容。
- 不允许处理业务导航逻辑。

### 9.2 MobileHeader

统一手机端顶部栏。

职责：

- 返回按钮。
- 页面标题。
- 微信胶囊避让。
- 安全区适配。
- 顶部背景和边界。

规则：

- 高度默认 44px。
- 标题使用 `text.navTitle`。
- 返回按钮点击热区不小于 44px。
- 不允许页面内部重复模拟状态栏。
- 如果使用统一小程序壳层，页面不得额外加一层顶部安全区。

### 9.3 IconBadge

统一图标徽章。

职责：

- 图标容器尺寸。
- 图标尺寸。
- 线宽。
- 圆角。
- 语义色。

允许参数：

```text
size: sm | md | lg | xl
tone: brand | success | warning | danger | ai | neutral
shape: rounded | circle
```

禁止：

- 不允许传入任意宽高。
- 不允许业务组件自己写图标容器样式。
- 不允许同组入口混用多种图标尺寸。

### 9.4 MobileCard

统一卡片。

允许变体：

```text
variant: flat | card | raised | hero | ai | metric
padding: sm | md | lg
```

规则：

- `flat`：无阴影，用于列表。
- `card`：普通白卡。
- `raised`：强调卡。
- `hero`：页面主视觉。
- `ai`：AI 结果卡。
- `metric`：数据统计卡。

禁止：

- 不允许业务页面直接复制卡片圆角和阴影。
- 不允许同一组卡片使用不同圆角和阴影。

### 9.5 MenuSection

用于“我的页面”、设置页、资料页等分组入口。

职责：

- 分组标题。
- 分组图标。
- 内部分隔线。
- 子项布局。

规则：

- 分组标题使用 `text.sectionTitle`。
- 分组图标使用 `icon.sm` 或 `icon.md`。
- 分组卡片使用 `radius.xl` 和 `shadow.card`。
- 同一页面内所有 `MenuSection` 视觉必须一致。
- 当 `MenuSection` 内部只有少量菜单项时，分组图标与菜单项图标应使用同一容器尺寸，避免同组入口出现明显比例割裂。

### 9.6 MenuItem

用于菜单入口。

职责：

- 左侧图标。
- 主标题。
- 辅助说明。
- 右侧箭头/状态。
- 点击反馈。
- 禁用/危险状态。

规则：

- 主标题使用 `text.itemTitle`。
- 说明使用 `text.helper`。
- 左侧图标默认使用 `icon.md`；只有功能宫格、强入口或大触控卡片才使用 `icon.lg`。
- 高度必须保证触控热区不小于 44px。
- 危险操作只改变颜色语义，不改变布局结构。
- 右侧 `value` 用于短辅助信息，例如手机号、版本号、状态、数量、日期。
- 下方 `description` 仅用于解释性说明、风险提示、多字段摘要，不用于短版本号或短状态值。
- 对用户已经能从标题理解的入口，不添加说明文案，避免降低信息密度。

### 9.7 MetricCard

用于数据统计。

职责：

- 指标名称。
- 核心数字。
- 说明文字。
- 趋势/状态。

规则：

- 数字使用 `text.metric`。
- 指标名使用 `text.label` 或 `text.helper`。
- 卡片使用 `radius.xl`。
- 同一组指标卡必须统一高度、圆角、阴影。

### 9.8 BottomSheet

用于底部抽屉。

职责：

- 遮罩。
- 顶部圆角。
- 标题栏。
- 关闭按钮。
- 内容滚动。
- 底部安全区。

规则：

- 顶部圆角使用 `radius.sheet`。
- 阴影使用 `shadow.modal`。
- 必须提供关闭路径。
- 关闭按钮点击热区不小于 44px。
- 内容超出时必须在抽屉内部滚动，不能让滚动条跑到外层页面。

### 9.9 ActionButton

统一按钮。

允许变体：

```text
variant: primary | secondary | ghost | danger
size: sm | md | lg
```

规则：

- 主要按钮高度不小于 44px。
- 普通按钮高度不小于 40px。
- 纯图标按钮热区不小于 44px。
- 必须提供 `active` 按压反馈。
- 禁用态必须同时降低颜色和交互反馈。

## 10. 交互状态

### 10.1 必须具备的状态

所有可交互组件必须考虑：

- 默认态。
- 按下态。
- 选中态。
- 禁用态。
- 加载态。
- 错误态。

### 10.2 触屏反馈

- 核心按钮使用 `active:scale-95` 或 `active:opacity-70`。
- 列表项可使用 `active:bg-slate-50`。
- 卡片型入口可使用轻微缩放，但不得造成布局跳动。
- 反馈时长建议 120ms - 150ms。
- 手机端核心交互不得依赖 `hover`（鼠标悬停）。

## 11. 手机端真实场景规则

### 11.1 教师记录页

目标：快速识别、快速修改、减少误触。

规则：

- 记录卡片应优先保证正文可读。
- AI 分析结果可以使用独立 `CardAiResult`，但必须限制在记录流场景。
- 时间、对象、得分等字段应形成固定字段布局。
- 编辑按钮、替换按钮、确认按钮必须有明确触控热区。
- 记录流卡片不得复制到普通列表页。

### 11.2 我的页面

目标：入口清晰、分组稳定、账号信息可信。

规则：

- 使用 `MenuSection` + `MenuItem`。
- 所有菜单图标使用统一 `IconBadge`。
- 菜单主标题不得使用过重字重。
- 分组卡片圆角、阴影、内边距必须统一。
- 个人资料 Hero 卡可以使用更高层级，但不能影响下方菜单项规范。
- 分组卡片之间默认使用 12px 间距；个人资料 Hero 卡与第一组菜单默认使用 16px 间距。
- 手机号、账号状态这类短信息放在菜单项右侧，作为辅助值展示。
- 版本号这类短值放在菜单项右侧，作为辅助值展示；版本说明、更新说明等长文本才放在下方。
- 明确入口如“我的文件”“使用督导报表”默认不写说明；只有存在误解风险或需要解释状态时才添加辅助说明。

### 11.3 报表页

目标：数字突出、结论清楚、阅读连续。

规则：

- 核心数字使用 `text.metric`。
- 指标卡使用 `MetricCard`。
- 图表说明使用 `text.helper`。
- 报告封面、导出预览类页面允许有独立排版，但必须封装在报告组件内。
- 报表页不得把海报式字号带回普通业务页面。

### 11.4 文件页/列表页

目标：扫描效率和操作稳定。

规则：

- 文件卡片使用 `CardListItem` 或 `MobileCard variant="card"`。
- 文件图标使用 `IconBadge` 或文件专用图标容器。
- 更多、删除等操作按钮必须保持 44px 热区。
- 弹出菜单使用统一浮层样式，不得单页写死。

### 11.5 手机端录入/确认页

目标：状态清晰、反馈明确、错误可恢复。

规则：

- 主操作必须唯一且突出。
- 输入状态、分析中状态、成功状态、错误状态必须有明确差异。
- 录音、拍照、文字入口需保持统一图标体系。
- 结果确认页应突出待确认内容，不应用过多装饰抢占注意力。

### 11.6 子页面顶部栏

目标：只保留页面导航语义，不重复模拟系统原生控件。

规则：

- 手机壳模式下，状态栏和微信胶囊由统一外层容器模拟，子页面不得再次绘制微信胶囊。
- 子页面顶部栏只保留返回按钮、页面标题和必要操作位。
- 如右侧没有业务操作，应保留等宽空占位以维持标题居中，不放置装饰性控件。
- 子页面不得在顶部重复添加状态栏安全区；安全区由统一手机壳或页面容器处理。
- 业务更多菜单可以使用更多图标，但必须是具体业务操作入口，不得伪装成微信原生胶囊。

## 12. 例外机制

### 12.1 允许例外的场景

以下场景允许偏离基础 token：

- 手机壳和微信原生模拟层。
- 报告封面、证书、海报。
- AI 分析结果卡。
- 空状态插画。
- 特殊演示型 Hero 区。

### 12.2 例外要求

任何例外必须满足：

- 有明确场景原因。
- 封装在语义组件内。
- 不得复制到普通业务页面。
- 不得破坏触控热区。
- 不得影响同类组件一致性。
- 出现 3 次以上必须升级为正式组件或 token。

---

## 13. 禁止事项

手机端页面禁止：

- 直接堆 `text-[13px]`、`rounded-[28px]`、`shadow-[...]` 这类一次性样式。
- 同一组卡片使用不同圆角、阴影、边框。
- 普通正文使用过多 `font-black`。
- 图标无容器、点击区域过小。
- 核心交互依赖 `hover`。
- 页面内部重复模拟状态栏。
- 业务页面复制另一页面的整段样式。
- 基础组件写死业务文案。
- 为单页视觉效果破坏通用组件结构。
- 未说明原因新增视觉 token。

---

## 14. 落地检查清单

每次新增或修改手机端页面，必须检查：

1. 是否使用现有字体角色？
2. 是否使用现有圆角 token？
3. 是否使用现有阴影等级？
4. 图标尺寸、线宽、容器是否统一？
5. 点击热区是否不小于 44px？
6. 是否有按压反馈？
7. 是否可以通过通用组件实现？
8. 是否存在业务样式侵入基础组件？
9. 是否存在可复用组件被复制粘贴？
10. 如有例外，是否已说明场景原因？

---

## 15. 专业取舍结论

本规范不追求“token 越少越好”，也不追求“场景覆盖越多越好”。正式原则是：

- 字体层级少，保证阅读秩序。
- 圆角层级少，保证系统感。
- 阴影层级少，保证层级稳定。
- 图标角色清晰，保证入口一致。
- 组件语义丰富，保证真实业务能复用。
- 例外机制严格，保证特殊页面不污染普通页面。

项目后续手机端 UI 应以本规范为准。若页面现状与规范冲突，应优先收敛到规范；若规范无法覆盖真实手机场景，应先补充规范或新增语义组件，再改页面。
