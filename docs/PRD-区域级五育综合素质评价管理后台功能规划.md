# 区域级五育综合素质评价管理后台功能规划

版本：V1.0  
日期：2026-06-02  
来源依据：`区域五育并举综合素质评价系统建设方案V1.docx`  
适用范围：区域教育局/区级管理后台，不包含学校端、教师端、家长端、学生端的完整功能设计；仅说明与区级管理后台强相关的功能边界。

---

## 1. 产品定位

区域级管理后台是教育局用于统一管理全区五育并举综合素质评价工作的核心后台。它负责统一评价标准、接入区域教育数据中台、监管各校评价进度、分析区域五育发展态势，并输出区域治理和督导决策所需的数据报告。

该后台不是单纯的数据看板，而是一个“区域评价治理平台”。它需要同时解决四类问题：

1. 全区评价标准不统一。
2. 学校评价数据分散，难以比较。
3. 教育局缺少过程监管和督导抓手。
4. 五育评价结果难以沉淀为区域治理依据。

---

## 2. 建设目标

### 2.1 业务目标

- 建立区级统一的德智体美劳评价指标体系。
- 支持学校在区级标准下扩展校本特色指标。
- 汇聚区域教育数据中台和学校过程性评价数据。
- 实现全区评价进度、数据质量、学校差异的动态监管。
- 形成区域、学校、年级、学段、五育维度的综合分析报告。
- 为教育局督导、资源配置、政策优化提供数据依据。

### 2.2 产品目标

- 让教育局能配置规则，而不是依赖技术人员改系统。
- 让区级管理员能看清全区评价工作是否真实开展。
- 让业务科室能快速定位薄弱学校、薄弱学段和薄弱维度。
- 让领导层能通过大屏和报告掌握区域五育发展态势。
- 让数据管理员能追溯每一类评价数据的来源、状态和质量。

### 2.3 成功指标建议

- 学校接入率：试点期不少于目标学校的 30%，全区推广期达到 100%。
- 指标启用率：区级核心指标启用率达到 100%。
- 数据同步成功率：中台数据同步成功率不低于 98%。
- 评价覆盖率：学生评价覆盖率不低于 95%。
- 进度监管有效性：系统可自动识别漏评、迟评、异常数据学校。
- 报告产出效率：区级月报、季报、年度报告可自动生成初稿。

## 3. 用户角色与权限边界

| 角色 | 主要职责 | 核心权限 |
|---|---|---|
| 区级超级管理员 | 管理全区系统配置和最高权限 | 组织管理、角色管理、系统参数、全量数据查看、审计日志 |
| 区级业务管理员 | 管理五育评价业务规则和评价工作 | 指标配置、评价周期、学校监管、报告查看、督导任务 |
| 区级数据管理员 | 负责数据中台对接和数据质量治理 | 数据源配置、同步任务、数据台账、异常处理、脱敏规则 |
| 区级领导/决策者 | 查看全区态势和决策报告 | 大屏查看、区域报告、学校对比、趋势分析 |
| 督导人员 | 跟进学校评价落实情况 | 进度查看、异常提醒、督导台账、整改跟进 |
| 学校管理员 | 在区级授权下维护本校配置 | 查看本校数据、提交校本指标、处理区级督导任务 |

说明：学校管理员可以登录区域平台的学校视角，但不应看到其他学校数据。

## 4. 后台一级菜单规划

建议区域级后台按 9 个一级模块规划：

1. 工作台
2. 区域数据大屏
3. 指标体系管理
4. 学校与组织管理
5. 数据中台与数据治理
6. 评价过程监管
7. AI 分析与成长画像
8. 报告中心与督导台账
9. 系统设置与安全审计

## 5. 功能模块规划

## 5.1 工作台

### 功能定位

工作台用于承载区级管理员的日常待办、风险提醒和关键数据概览，是进入后台后的默认首页。

### 核心功能

| 功能 | 说明 | 优先级 |
|---|---|---|
| 今日概览 | 展示学校接入数、学生覆盖数、评价完成率、数据异常数 | P0 |
| 待处理事项 | 包括校本指标待审核、异常数据待处理、督导任务待跟进 | P0 |
| 风险预警 | 展示漏评、迟评、数据同步失败、异常评分波动 | P0 |
| 快捷入口 | 快速进入指标配置、数据同步、评价监管、报告生成 | P1 |
| 最近报告 | 展示最近生成的区级月报、学校对比报告、督导报告 | P1 |

### 关键设计要求

- 首页必须区分“数据概览”和“待办处理”，避免只做展示不支持行动。
- 风险提醒需要可点击追溯到具体学校、班级、指标或数据源。

---

## 5.2 区域数据大屏

### 功能定位

区域数据大屏用于给教育局领导和业务科室查看全区五育发展态势，重点服务汇报、监管和决策。

### 核心功能

| 功能 | 说明 | 优先级 |
|---|---|---|
| 全区五育总览 | 展示德智体美劳五大维度综合发展情况 | P0 |
| 学校分布地图/列表 | 按学校展示评价完成率、综合水平、风险状态 | P0 |
| 五育维度趋势 | 按月、学期、学年查看各维度变化趋势 | P0 |
| 校际对比 | 对比学校综合水平、单项维度、数据质量 | P0 |
| 学段/年级分析 | 查看小学、初中、高中或各年级差异 | P1 |
| 薄弱维度识别 | 自动识别区域短板维度和薄弱学校 | P1 |
| 大屏展示模式 | 支持无操作展示、轮播、全屏汇报 | P1 |

### 数据维度

- 区域维度：全区、片区、街道、集团校。
- 学校维度：学校类型、学校名称、办学层级。
- 学生维度：学段、年级、班级、性别等，涉及隐私时仅做聚合展示。
- 时间维度：日、周、月、学期、学年。
- 五育维度：德育、智育、体育、美育、劳育。

### 关键设计要求

- 大屏展示聚合数据，不直接暴露学生个人隐私。
- 所有排名类展示应支持关闭或脱敏，避免形成简单排名导向。
- 区域短板识别要展示依据，不能只给结论。

---

## 5.3 指标体系管理

### 功能定位

指标体系管理是后台的核心模块，用于统一全区五育评价标准，并允许学校在授权范围内补充校本特色指标。

### 核心功能

| 功能 | 说明 | 优先级 |
|---|---|---|
| 区级指标库 | 建立德智体美劳一级维度、二级观测点、三级评价标准 | P0 |
| 指标权重配置 | 配置不同维度、学段、周期下的权重 | P0 |
| 评分规则配置 | 支持等级、分值、达标、不达标、加减分等规则 | P0 |
| 适用范围配置 | 指标可按学段、年级、学校类型启用 | P0 |
| 校本指标授权 | 设置学校是否可新增、修改或停用校本指标 | P0 |
| 校本指标审核 | 学校提交特色指标后，由区级审核备案 | P0 |
| 指标版本管理 | 支持版本发布、历史版本查看、变更记录 | P1 |
| 指标模板管理 | 提供德育、体育、美育、劳育等常用模板 | P1 |

### 指标结构建议

每个指标至少包含：

- 指标名称。
- 所属五育维度。
- 观测点说明。
- 评分方式。
- 数据来源。
- 适用学段。
- 评价周期。
- 权重。
- 是否区级强制。
- 是否允许学校扩展。
- 是否纳入区域分析。

### 关键设计要求

- 区级核心指标不可被学校删除或篡改。
- 校本指标要能备案、审核、退回和停用。
- 指标变更必须保留历史版本，避免影响历史评价结果。

---

## 5.4 学校与组织管理

### 功能定位

该模块用于维护区、校、年级、班级、教师、学生等基础组织关系，是权限控制和数据归属的基础。

### 核心功能

| 功能 | 说明 | 优先级 |
|---|---|---|
| 学校管理 | 维护学校名称、代码、类型、学段、区域归属 | P0 |
| 组织架构同步 | 从数据中台同步年级、班级、学生、教师基础信息 | P0 |
| 学校账号管理 | 开通学校管理员账号，重置密码，停用账号 | P0 |
| 角色授权 | 按角色分配菜单权限、数据权限、操作权限 | P0 |
| 学校接入状态 | 展示各校接入、配置、数据同步、评价开展状态 | P0 |
| 学校分组 | 支持按片区、集团校、试点批次分组管理 | P1 |

### 关键设计要求

- 组织数据应优先从区域数据中台同步，减少重复维护。
- 学校信息变更要有来源标识，区分“中台同步”和“后台手动维护”。
- 区级用户可查看全区，学校用户只能查看本校。

---

## 5.5 数据中台与数据治理

### 功能定位

该模块负责接入区域教育数据中台，并对五育相关数据进行同步、清洗、校验、脱敏和追溯。

### 核心功能

| 功能 | 说明 | 优先级 |
|---|---|---|
| 数据源管理 | 配置中台接口、数据库、文件同步等数据来源 | P0 |
| 同步任务管理 | 设置同步频率、同步范围、失败重试 | P0 |
| 数据范围配置 | 配置德智体美劳各类数据的接入范围 | P0 |
| 数据质量校验 | 检查缺失、重复、异常、口径不一致数据 | P0 |
| 数据异常处理 | 异常数据进入待处理列表，支持修正、忽略、退回 | P0 |
| 数据台账 | 记录数据来源、同步时间、责任主体、变更记录 | P0 |
| 数据脱敏规则 | 对学生姓名、证件号、联系方式等敏感信息脱敏 | P0 |
| 数据回流 | 将学校过程性评价结果汇总回区级数据池 | P1 |

### 五育数据范围建议

| 维度 | 典型数据 | 主要来源 |
|---|---|---|
| 德育 | 奖惩记录、志愿服务、文明评比、荣誉表彰 | 数据中台、学校上报 |
| 智育 | 学业成绩、学科素养、竞赛、学业进步 | 数据中台、教务系统 |
| 体育 | 体质健康、体育达标、运动打卡、赛事参与 | 数据中台、体育系统 |
| 美育 | 艺术测评、作品、活动参与、课程学习 | 学校上报、活动系统 |
| 劳育 | 校园劳动、社会实践、家务劳动、劳动课程 | 学校上报、家校端 |

### 关键设计要求

- 数据治理必须有“来源、时间、责任人、处理状态”。
- AI（人工智能）清洗只能作为辅助，关键修正需要人工确认。
- 所有数据同步失败都应进入告警和台账。

## 5.6 评价过程监管

### 功能定位

评价过程监管用于让教育局实时掌握各学校评价工作是否按要求开展，重点解决“有没有评、评得是否及时、数据是否异常”的问题。

### 核心功能

| 功能 | 说明 | 优先级 |
|---|---|---|
| 评价周期管理 | 配置月度、学期、学年等评价周期 | P0 |
| 学校进度监控 | 查看各校评价完成率、覆盖率、迟评情况 | P0 |
| 指标落地监控 | 查看区级核心指标在各校是否启用和执行 | P0 |
| 漏评迟评预警 | 自动识别未评价、逾期评价、覆盖率过低 | P0 |
| 异常评分预警 | 识别评分突增、突降、全员满分等异常情况 | P1 |
| 督办提醒 | 向学校发送整改提醒或督办任务 | P0 |
| 整改跟踪 | 查看学校整改状态、反馈说明、复核结果 | P0 |

### 关键设计要求

- 监管不是简单排名，应突出问题定位和整改闭环。
- 每条预警需要说明触发规则，便于学校理解和申诉。
- 督办任务应支持状态流转：待处理、处理中、已反馈、已复核、已关闭。

## 5.7 AI 分析与成长画像

### 功能定位

该模块用于基于全区五育数据进行智能分析，形成学生、班级、学校、区域多层级画像。区级后台重点使用学校画像和区域画像，不直接过度展示学生个人细节。

### 核心功能

| 功能 | 说明 | 优先级 |
|---|---|---|
| 区域五育画像 | 生成全区五育发展结构和趋势 | P0 |
| 学校画像 | 展示学校五育优势、短板、变化趋势 | P0 |
| 学段画像 | 按小学、初中、高中或年级分析差异 | P1 |
| 薄弱点识别 | 识别区域、学校、学段的薄弱指标 | P0 |
| 智能建议 | 生成整改建议、资源配置建议、督导建议 | P1 |
| 模型解释 | 展示分析依据、参与指标、数据来源 | P1 |
| 人工复核 | 对 AI 生成结论支持人工确认、编辑、废弃 | P0 |

### 关键设计要求

- AI 结论必须可解释，不能只给“优秀/薄弱”等标签。
- AI 建议不能直接作为行政结论，应标注“辅助分析”。
- 涉及学生个人画像时，区级后台默认聚合展示，必要时走授权查看。

## 5.8 报告中心与督导台账

### 功能定位

该模块用于自动生成区级、学校级、专题类报告，并沉淀督导检查过程资料。

### 核心功能

| 功能 | 说明 | 优先级 |
|---|---|---|
| 区级综合报告 | 自动生成月度、季度、年度五育评价报告 | P0 |
| 学校对比报告 | 对比学校五育发展、评价完成、数据质量 | P0 |
| 专题分析报告 | 生成体育、美育、劳动教育等专题报告 | P1 |
| 督导台账 | 记录督导事项、问题、整改、复核、归档 | P0 |
| 报告模板管理 | 配置报告结构、指标、图表和导出格式 | P1 |
| 报告导出 | 支持导出 Word、PDF、Excel 等格式 | P0 |
| 报告留档 | 历史报告归档，支持按时间、学校、主题查询 | P0 |

### 报告类型建议

- 全区五育发展月报。
- 学期综合素质评价报告。
- 学校评价工作落实报告。
- 学校五育质量对比报告。
- 薄弱学校整改跟踪报告。
- 数据质量与评价覆盖率报告。

### 关键设计要求

- 报告要支持“一键生成初稿 + 人工编辑确认”。
- 导出报告应保留生成时间、数据周期、统计口径。
- 督导台账要与评价过程监管预警打通。

## 6. 关键业务流程

## 6.1 区级指标发布流程

1. 区级业务管理员创建或编辑五育指标。
2. 设置适用学段、评价周期、权重、数据来源。
3. 保存为草稿。
4. 提交审核或内部确认。
5. 发布指标版本。
6. 学校接收区级指标。
7. 学校在授权范围内补充校本指标。
8. 区级审核校本指标。
9. 指标正式用于评价和分析。

## 6.2 数据同步与治理流程

1. 区级数据管理员配置数据源。
2. 系统按计划同步中台数据。
3. 系统进行数据校验和清洗。
4. 正常数据进入五育数据池。
5. 异常数据进入待处理列表。
6. 数据管理员处理异常。
7. 处理结果进入数据台账。
8. 数据参与评价建模和报告生成。

## 6.3 评价监管与督办流程

1. 系统按评价周期统计学校完成情况。
2. 自动识别漏评、迟评、异常评分、数据缺失。
3. 区级业务管理员查看预警原因。
4. 向学校发起督办任务。
5. 学校反馈整改说明和处理结果。
6. 区级复核整改结果。
7. 督办事项关闭并进入台账。

## 6.4 区域报告生成流程

1. 选择报告类型和统计周期。
2. 系统拉取相关指标、学校、数据源。
3. 自动生成图表、摘要和分析建议。
4. 区级人员人工编辑和确认。
5. 导出报告。
6. 报告归档。

## 7. 数据对象规划

区域级后台至少需要管理以下核心数据对象：

| 数据对象 | 说明 |
|---|---|
| 区域组织 | 区县、片区、集团校等组织层级 |
| 学校 | 学校基础信息、类型、学段、接入状态 |
| 用户与角色 | 区级管理员、学校管理员、督导人员等 |
| 学生基础信息 | 来源于数据中台，区级默认聚合使用 |
| 教师基础信息 | 来源于数据中台或学校同步 |
| 五育指标 | 德智体美劳指标、评分规则、权重、版本 |
| 校本指标 | 学校扩展指标、审核状态、适用范围 |
| 评价记录 | 学校过程性评价数据和中台客观数据 |
| 数据同步任务 | 数据源、同步时间、成功率、失败原因 |
| 数据异常 | 缺失、重复、冲突、异常波动等 |
| 预警事件 | 漏评、迟评、异常评分、数据异常 |
| 督办任务 | 问题、责任学校、整改状态、复核结果 |
| 分析报告 | 区域报告、学校报告、专题报告 |
| 操作日志 | 登录、修改、导出、审批、删除等行为 |

## 8. MVP 范围建议

MVP（最小可用版本）建议优先保证“能统一标准、能接数据、能看进度、能出报告”。

### P0：首期必须做

- 工作台基础概览。
- 区级指标库。
- 指标权重和评分规则配置。
- 学校与组织管理。
- 区域数据中台基础对接。
- 数据同步任务和数据台账。
- 学校评价进度监管。
- 漏评、迟评、数据异常预警。
- 区级综合报告生成。
- 学校对比报告。
- 角色权限和操作日志。

### P1：第二阶段增强

- 区域数据大屏全屏展示。
- 校本指标审核与版本管理。
- AI 薄弱点识别。
- 智能整改建议。
- 督导台账闭环。
- 报告模板管理。
- 学段、年级、片区多维分析。

### P2：后续扩展

- AI 模型持续学习。
- 个性化政策建议。
- 多区域横向对标。
- 与积分兑换终端打通。
- 与家长端、学生端成长反馈联动。
- 更细粒度的隐私授权和数据审批流程。

## 9. 与智能积分兑换终端的衔接建议

如果本项目要和“校园智能积分兑换终端”结合，区域级后台需要预留以下能力：

| 衔接点 | 说明 | 建议优先级 |
|---|---|---|
| 积分规则映射 | 将五育评价结果映射为积分或成长值 | P1 |
| 区级积分监管 | 查看各校积分发放、消耗、异常情况 | P1 |
| 兑换资源配置 | 区级可配置统一兑换资源或活动权益 | P2 |
| 终端数据回流 | 兑换、查询、核销行为回流区域后台 | P1 |
| 激励效果分析 | 分析积分兑换对学生参与和成长的影响 | P2 |
| 风险控制 | 防止刷分、异常兑换、超额发放 | P1 |

建议把积分终端定位为五育评价结果的“线下激励触点”，而不是孤立的兑换设备。

## 10. 风险与注意事项

### 10.1 业务风险

- 五育评价容易变成简单打分，需要避免过度排名。
- 学校执行差异大，必须有督办和培训机制。
- 校本特色指标如果不受控，可能导致区域数据不可比。

### 10.2 数据风险

- 中台数据字段和口径不统一，会影响评价准确性。
- 学校过程性数据主观性强，需要抽查和复核机制。
- 学生隐私数据敏感，导出和查看必须严格控制。

### 10.3 AI 风险

- AI 建议不能直接作为行政评价结论。
- AI 识别和分析必须展示依据。
- 对学生个人的评价建议需要人工确认，避免误判。

### 10.4 实施风险

- 原方案中 6-9 周全区落地周期偏紧。
- 建议先选 3-5 所学校试点，稳定后再扩展。
- 数据中台对接应作为项目最大不确定项提前验证。

## 11. 待确认问题

1. 区域后台是否只服务教育局，还是学校管理员也从同一后台进入？
2. 是否已有区域教育数据中台接口规范？
3. 五育指标是否已有政策文件或地方标准？
4. 是否需要把综合素质评价结果转换为积分？
5. 区级是否需要直接查看学生个人档案？如需要，授权边界是什么？
6. 报告是否要求固定 Word/PDF 格式？
7. 是否需要支持多个区县或只支持单一区域？
8. 是否要接入现有校园智能积分兑换终端？

---

## 12. 结论

区域级管理后台的核心不应只是展示数据，而应围绕“标准制定、数据治理、过程监管、区域研判、督导闭环”建设。首期应优先做稳指标体系、数据中台对接、评价进度监管和报告中心；AI 分析、大屏展示和积分终端联动可作为第二阶段增强能力。
