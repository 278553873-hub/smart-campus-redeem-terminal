# PRD-校长助理：周建议、月复盘与学期学校报告

> 版本：V1.2
> 更新日期：2026-07-20  
> 适用端：教师手机端 / 学校管理角色  
> 当前阶段：完整Demo + 后端协议定义  
> 对话能力：当前不开放，未来接入小龙虾

## 1. 摘要

校长助理提供三个固定周期的AI报告：

1. `本周管理建议`：分析上一个完整自然周，给出本周最需要推进的管理动作。
2. `上月学校复盘`：分析上一个完整自然月，判断持续问题、改善进展和下月重点。
3. `学期学校报告`：分析后台当前学期完整周期，生成类似学校学期运营汇报的正式报告。

系统每天更新学校数据快照、统计指标和异常候选，但不每天调用大模型生成报告。校长首次打开某一周期报告时才发起AI任务；同一学校、同一报告周期只保留一份正式结果，重复进入读取缓存。

当前版本不展示输入框、推荐问题或聊天入口。未来接入小龙虾后，从报告编号、数据快照和证据索引继续追问，按需查询明细，不把全校数据一次性写入对话上下文。

## 2. 问题

学校数据报表能够回答评价数量、学生覆盖、教师活跃、班级差异和指标分布等事实问题，但校长仍需要进一步理解：

- 本周最先处理什么。
- 哪些变化只是短期波动，哪些已经持续。
- 哪些做法值得在年级或全校复制。
- 上一次管理动作是否带来改善。
- 学期形成了什么成果、还存在哪些风险。
- 下一周期应由谁采取什么行动，用什么指标检查结果。

如果每天自动生成完整AI报告，会产生大量无人阅读的模型调用；如果把全校明细全部放进Prompt，又会造成上下文过长、成本高、数字错误和隐私风险。

## 3. 目标与非目标

### 3.1 目标

1. 校长在3分钟内看懂周建议或月复盘。
2. 周建议最多给出3条本周行动，每条包含责任角色和检查指标。
3. 月复盘同时解释趋势、上期动作结果和下月重点。
4. 每个学期形成一份结构完整、可追溯的正式学校报告。
5. 无人查看时不产生AI报告调用成本。
6. 同一周期不因重复点击或多人同时进入而重复生成。
7. 图表和精确数字由后台确定性生成，AI只负责解释和建议。
8. 为未来小龙虾对话保存报告、快照和证据上下文。

### 3.2 非目标

- 当前不提供自由对话、语音追问和推荐问题。
- 当前不让校长手动选择任意分析日期。
- 当前不根据记录量评价教师能力、态度、责任心或教学质量。
- 当前不生成公开分享链接。
- 当前Demo不接真实模型任务和真实学校数据库。

## 4. 用户与权限

首版支持：

- 学校管理员。
- 校长或被授权的学校领导。

普通教师、班主任和个人空间用户不展示校长助理入口。服务端在创建任务、查看报告和导出报告时均重新校验学校角色，不能只依赖前端隐藏入口。

## 5. 三类报告定位

| 报告 | 数据周期 | 校长要解决的问题 | 建议长度 |
|---|---|---|---:|
| 本周管理建议 | 上一个完整自然周 | 本周优先推动什么 | 700至1000字 |
| 上月学校复盘 | 上一个完整自然月 | 哪些改善有效、哪些问题持续 | 1000至1500字 |
| 学期学校报告 | 当前学期开始日至结束日 | 形成正式学期运营汇报 | 按章节生成 |

三类报告共享同一数据快照与证据体系，但不是同一篇文案的长短版本：

- 周报告强调优先级和立即行动。
- 月报告强调趋势、延续性和动作结果。
- 学期报告强调完整总结、代表实践、问题边界和下学期机制。

学期报告不是周报和月报正文的简单拼接。周月报告中的管理动作、改善结果和证据编号可作为学期候选材料，但学期生成时必须重新核验基础统计与原始证据。

## 6. 入口与页面

入口：教师手机端 → 我的 → 管理工具 → 校长助理。

首页固定展示：

1. `本周管理建议`：最高视觉权重。
2. `上月学校复盘`：次要入口。
3. `学期学校报告`：低频入口。

三个入口均可点击进入生成或报告阅读页，不出现“能力建设中”等占位状态。

### 6.1 往期报告入口

周、月、学期报告生成完成后，正式报告页右上角均展示历史图标：

- 周报告进入`往期管理建议`。
- 月报告进入`往期学校复盘`。
- 学期报告进入`往期学期报告`。

历史入口触控区域不小于44×44px，并提供明确的无障碍名称。生成等待页不展示历史入口，避免离开当前生成反馈；正式任务在服务端继续执行的规则不变。

### 6.2 历史列表与详情

- 三类历史报告分别展示，不混在同一列表。
- 报告按数据周期倒序排列，并按自然月或学年分组。
- 历史卡片只展示报告周期、数据周期和生成日期，不展示摘要、统计、状态标签或重新生成按钮。
- 只展示`generated`状态的正式报告；失败和生成中的任务不进入历史。
- 点击历史项进入完整报告详情，复用当前报告的同一阅读组件和内容结构。
- 历史详情不重复展示历史入口。
- 返回层级固定为：`历史详情 → 历史列表 → 当前报告`。
- 历史报告始终读取生成时保存的内容和数据快照，不随补录、学期配置或Prompt版本变化自动改写。
- 已生成的学期报告即使超过当前生成窗口，仍可从历史列表查看。
- 切换学校空间后重新查询该学校的历史列表，不能沿用上一学校的报告。

当前版本不展示：

- 对话输入框。
- “你可以这样问”等推荐问题。
- 禁用状态的聊天入口。
- Prompt、Token、数据表名称等实现说明。

## 7. 周报告规则

### 7.1 数据周期

固定读取上一个完整自然周，周一00:00至周日23:59:59。当前尚未结束的自然周不进入本次报告。

示例：2026年7月20日进入，报告数据周期为2026年7月13日至7月19日，报告用于指导7月20日至7月26日的管理动作。

### 7.2 生成规则

- 每周一开始可生成上一完整周报告。
- 用户首次进入时创建任务，不在周一自动调用AI。
- 同一学校、同一行动周只生成一份正式结果。
- 生成后本周新增数据不更新本周报告，进入下周报告数据源。
- 上周完全无有效数据时不调用AI，展示“上周暂无可分析数据”。
- 数据较少时允许生成受限报告，但必须明确证据不足，减少结论数量。

### 7.3 内容结构

1. 学校、数据周期和行动周期。
2. 上周核心数据。
3. 本周优先判断。
4. 上周关键信号：改善、风险或待核实事项。
5. 本周管理动作：最多3条，包含责任角色和检查指标。
6. AI生成说明。

## 8. 月报告规则

### 8.1 数据周期

固定读取上一个完整自然月。当前自然月尚未结束的数据不进入本次月报。

示例：2026年7月进入，报告数据周期为2026年6月1日至6月30日。

### 8.2 生成规则

- 每月1日起可生成上一个完整自然月报告。
- 用户首次进入时创建任务，不在月初自动调用AI。
- 同一学校、同一复盘月只生成一份正式结果。
- 报告生成后补录或修改数据不自动重写正式报告。
- 上月完全无有效数据时不调用AI，展示“上月暂无可分析数据”。
- 月报必须读取上一个月报中的管理动作及本月结果；不存在上月报告时明确为首次复盘。

### 8.3 内容结构

1. 学校和复盘月份。
2. 月度核心数据及环比变化。
3. 月度总体判断。
4. 改善进展及证据。
5. 持续问题及证据边界。
6. 下月管理动作：最多3条。
7. AI生成说明。

## 9. 学期报告规则

### 9.1 后台配置

后台“学期管理”至少提供：

```text
term_id
term_name
start_date
end_date
is_current
```

`start_date`和`end_date`决定统计范围，`end_date`决定生成窗口。

### 9.2 生成窗口

仅允许在以下两个自然月内创建新的学期报告：

1. 学期结束日期所在月份的前一个月。
2. 学期结束日期所在月份。

生成窗口从“结束月份前一个月的1日”开始，到“结束月份最后一天”结束。

```text
学期结束日期：2026-07-15
允许生成：2026-06-01 至 2026-07-31

学期结束日期：2027-01-20
允许生成：2026-12-01 至 2027-01-31
```

结束日期具体是月中还是月末，不改变按自然月计算的规则。

### 9.3 点击结果

| 状态 | 点击结果 |
|---|---|
| 已存在成功报告 | 直接打开，不受当前日期限制 |
| 未生成且在生成窗口内 | 创建任务并进入生成页 |
| 未生成且早于生成窗口 | 提示最早开放日期，不调用AI |
| 未生成且晚于生成窗口 | 提示管理员确认当前学期配置 |
| 学期时间无效 | 提示检查学期配置，不调用AI |

未到时间时：

```text
标题：暂未到生成时间
正文：{学期名称}报告将于{开放日期}起开放生成。
按钮：知道了
```

### 9.4 内容结构

学校运营版至少包含：

1. 学校名称、报告名称、周期和生成日期。
2. 学期核心数据。
3. 总体判断。
4. 班级与教师使用情况。
5. 优秀成果与高频使用样本。
6. 代表性班级实践或学生成长材料。
7. 五育与指标体系观察。
8. 需要关注的问题及证据边界。
9. 下学期深化应用建议。

附件中的系统服务升级、采购建议和供应商服务承诺不属于校长助理自动报告的默认内容；只有后台明确提供运营服务数据时，才能作为独立附录生成。

## 10. 统一生成流程

1. 用户点击报告入口。
2. 后端校验权限、学校空间、目标周期和已有报告。
3. 已生成时直接返回报告。
4. 未生成时冻结数据水位线，生成结构化快照。
5. 后端计算统计、趋势、分布和候选洞察。
6. 后端按候选问题定向筛选代表记录。
7. AI按照固定JSON结构生成报告。
8. 后端完成数字一致性、对象一致性和证据完整性校验。
9. 保存正式报告、数据快照和证据索引。
10. 前端显示连续阅读内容；重复进入读取缓存。

等待页只展示真实的任务阶段名称，不展示虚假百分比。用户离开页面后任务继续，再次进入读取同一任务状态。

## 11. AI输入数据协议

### 11.1 输入结论

Prompt输入不是全校所有原始明细，也不是把“学校数据报表”页面转成一段文字。正式输入由后台以JSON提供：

```text
学校与周期信息
+ 确定性核心统计
+ 时间趋势与对比
+ 年级、班级、教师紧凑统计
+ 五育与指标使用结构
+ 后端识别的异常与亮点候选
+ 上期管理动作及本期结果
+ 定向筛选的代表记录
+ 每项结论可引用的证据编号
+ 输出结构与安全约束
```

学校数据报表中的统计字段是主要数据来源之一，但必须读取后台结构化字段，不能让AI从截图、表格文本或长记录中重新计数。

### 11.2 确定性统计

至少包含：

- 原始评价事件总数、展开后的学生明细数，两者分开统计。
- 学生覆盖人数、覆盖率、长期未进入记录视野人数。
- 教师总数、活跃教师数、活跃天数和连续参与情况。
- 年级、班级评价量、覆盖率、参与教师数和最近记录时间。
- 五育正负向事件、净得分和事件分布。
- 二级、三级指标覆盖率、零覆盖和低频指标。
- 周趋势、月趋势、环比和班级差异分布。
- 已提出管理动作、责任角色、处理状态和干预后变化。

所有精确数量、比例、排序和图表由后台计算，AI不得自行计算或修改。

### 11.3 代表记录

后端先使用统计定位候选，再按班级、教师、学生、场景、时间和正负向分层筛选证据。

- 不随机抽样。
- 不只保留高分、负向或高频对象。
- 全班评价只发送一次原始事件，不按学生人数重复。
- 不发送原始语音、图片，只发送确认后的结构化文本。
- 周报告建议最多60条代表记录。
- 月报告建议最多100条代表记录。
- 学期报告每个章节建议最多120条代表记录。
- 超出预算时先按章节提取证据，不扩大单次Prompt。
- 每条AI结论必须保存统计字段或记录编号。

### 11.4 三类Prompt文件

- `docs/校长助理_本周管理建议_内容生成提示词.md`
- `docs/校长助理_上月学校复盘_内容生成提示词.md`
- `docs/校长助理_学期学校报告_内容生成提示词.md`

## 12. AI生成与校验策略

### 12.1 周报告

一次调用返回固定JSON。后台校验行动不超过3条、每条包含责任角色和检查指标、所有数字可回查。

### 12.2 月报告

一次调用返回固定JSON。后台校验环比方向、上期动作状态和本期证据一致，不存在上月报告时不得虚构历史动作。

### 12.3 学期报告

不使用一个超大Prompt一次生成：

1. 生成统一数据快照和候选清单。
2. 按“总体概况、使用情况、亮点、实践、指标、问题、行动”分别生成章节JSON。
3. 对每章进行数字、对象和证据校验。
4. 使用只包含章节结果的编辑Prompt统一语气、去重和排序。
5. 合并为手机连续阅读内容，后续Word或PDF复用同一结构化内容。

### 12.4 共同约束

- 只依据提供的数据和证据。
- 不把评价记录等同于学校真实事件的完整分布。
- 不给学生贴标签。
- 不根据记录量推断教师能力、态度、责任心或教学质量。
- 不把低频使用直接解释为“不重视”，只能提出核实方向。
- 没有证据时明确数据不足，不补写原因。
- 每条行动包含责任角色和后续检查指标。
- 每份报告显示“由AI基于学校评价数据生成，仅供管理与复盘参考”。

## 13. 状态、缓存与唯一键

任务状态：

```text
not_started
insufficient_data
queued
generating
generated
failed
```

唯一键：

```text
周报告：school_id + action_week_start + weekly_advice
月报告：school_id + review_month + monthly_review
学期报告：school_id + term_id + term_report
```

每份成功报告至少保存：

```text
report_id
school_id
report_type
period_key
status
data_period_start
data_period_end
generated_at
data_snapshot_id
data_watermark
prompt_version
content_version
content
evidence_refs
```

同一唯一键只允许一个生成任务。并发点击复用同一任务；普通用户不提供重新生成。数据错误修复、输出解析失败或安全拦截时，由系统或管理员在原唯一键下生成新版本。

历史列表接口按`school_id + report_type`查询，只返回报告编号、周期、生成日期和内容版本等列表字段；进入详情后再按`report_id`读取完整内容。服务端每次查询列表或详情时均重新校验学校权限。

## 14. 未来小龙虾对话

当前不开放对话，但每份正式报告提前保存：

- 报告编号。
- 数据快照编号。
- 洞察对应的证据编号。
- 可继续追问的主题。
- 当前用户可访问的学校、年级和学生范围。

未来小龙虾从`report_id`开始，只读取报告摘要。校长继续追问时，再按权限查询年级、班级、教师或记录证据，不把全部学校数据写入初始上下文。

## 15. 异常与边界

| 场景 | 处理方式 |
|---|---|
| 周或月完全无有效数据 | 不调用AI，展示无可分析数据 |
| 数据较少 | 减少结论，明确证据不足 |
| 后台未设置当前学期 | 提示管理员完成学期配置 |
| 学期日期格式错误 | 不调用AI，提示检查配置 |
| 生成过程中离开 | 任务继续，再次进入复用任务 |
| 生成失败 | 系统最多自动重试2次 |
| 输出解析失败 | 不展示原始模型文本，重试或失败 |
| 已生成后超过窗口 | 仍允许查看已有报告 |
| 切换学校空间 | 重新按新学校权限和唯一键检查 |
| 权限被移除 | 立即禁止查看当前和历史报告 |
| 学期时间被修改 | 未生成按新时间判断；已生成保留原周期和版本 |

## 16. 验收标准

### 16.1 页面与交互

- 首页展示`本周管理建议`、`上月学校复盘`、`学期学校报告`三个入口。
- 三个入口都可进入生成和正式报告页面，不出现占位提示。
- 当前版本不存在输入框、聊天入口和推荐问题。
- 周、月、学期等待页均展示明确任务阶段，不展示虚假百分比。
- 生成完成后显示完整报告和AI使用边界。
- 同一应用会话重复进入直接打开已生成结果。
- 周、月、学期报告生成完成后，正式报告右上角均可进入对应历史列表。
- 历史列表按周期倒序，仅展示周期和生成日期等必要信息。
- 历史详情复用正式报告阅读页，且不重复展示历史入口。
- 历史详情返回历史列表，历史列表返回当前报告。
- 所有触控热区不小于44×44px，并保留键盘焦点样式。
- 学期未到生成时间时使用统一底部提示，并显示最早开放日期。
- 周报正文突出不超过3条本周动作。
- 月报展示环比、改善进展和持续问题。
- 学期报告包含使用情况、代表实践、五育观察、问题和下学期建议。

### 16.2 业务规则

- 每天只更新数据与候选，不自动生成AI报告。
- 周报只读取上一个完整自然周。
- 月报只读取上一个完整自然月。
- 学期报告遵守结束月份及其前一个月的生成窗口。
- 窗口外不创建学期AI任务。
- 同一学校、同一周期并发请求只创建一个任务。
- 已生成报告重复访问读取缓存。
- 历史列表只返回当前学校、当前用户有权查看的成功报告。
- 历史报告固定读取生成时的数据快照和内容版本，不自动重写。
- 后台精确统计与AI解释职责分离。

### 16.3 内容与安全

- 三份提示词文档均包含输入协议、固定JSON输出和安全约束。
- 报告结论均有关联证据。
- 不以记录数量直接评价教师工作质量、态度或责任心。
- 不把学生未出现于记录解释为没有表现或存在问题。
- 不推断学生心理、医学、家庭、人格或智力信息。
- 推测性判断明确使用“建议核实”并说明核实方向。
- 正式报告明确标注AI生成和使用边界。
