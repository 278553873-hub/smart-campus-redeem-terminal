# PRD-班主任助理：本周行动建议与我的评价复盘

> 版本：V1.2
> 更新日期：2026-07-16
> 适用端：教师手机端  
> 当前阶段：领导汇报 Demo + 后续正式开发规则  
> 文档状态：待产品、研发、AI 服务与测试共同评审

## 1. 摘要

班主任助理提供两项不同周期的AI能力。“本周行动建议”读取班级上一个完整自然周的全部教师评价记录，帮助班主任确定本周行动；“我的评价复盘”读取班主任本人在上一个完整自然月的评价记录，结合班级名单和学校指标体系，发现个人观察与记录方式中的盲区。两类报告均按班级生成并支持查看历史。

本周行动建议数据不足时不向前追溯，也不使用本周补录数据生成本周建议；我的评价复盘数据不足时不调用AI，只展示记录量和报告示例。两项能力均采用首次访问时生成、成功后缓存，不自动重复生成。

当前 Demo 用于展示页面与内容效果，不代表已经接入真实班级数据和AI模型。正式版本需要按本文的数据、权限、生成、缓存和异常规则开发。

## 3. 背景

### 3.1 用户问题

班主任每天会看到不同老师提交的学生评价记录，但很难在有限时间内完成以下工作：

1. 从分散记录中找到连续出现的学生信号。
2. 判断哪些问题是个别事件，哪些是班级共性。
3. 关注其他老师发现、但班主任可能没有看到的学生亮点。
4. 把观察转化为少量、明确、可执行的班级管理动作。
5. 从评价明细中理解各位教师在本班的评价参与度、鼓励与纠偏方式、评价对象粒度和观察侧重，而不只看到次数排名。
6. 判断自己的记录是否长期集中在少数学生或明显事件上。
7. 发现记录中把观察事实过早写成态度、责任或品格判断的情况。
8. 结合学校指标体系，发现指标错配、过度集中和可以自然补充的观察视角。

已有统计报表适合查看数量和趋势，但不能直接回答“本周应该先做什么”。

本周行动建议中的“评价洞察”也不能停留在“某位老师评价多少次、占比多少、覆盖多少学生”。它需要结合原始评价事件和明细，进一步解释教师评价参与是否集中、评价以鼓励还是纠偏为主、评价主要面向个人还是全班、行为描述是否具体，以及不同教师提供的观察视角是否形成互补。

统计报表也不能解释“为什么这些学生更容易进入记录”“教师通常在什么事件发生后才开始记录”“行为描述是否足以支持所选指标”。我的评价复盘必须基于评价明细提供解释性洞察，不能重新制作一份数量报表。

### 3.2 为什么固定分析上一个完整自然周

本功能的目标不是兼容长期不记录的班级，而是让老师明确感受到“持续记录，下一周获得行动建议”的价值。因此固定时间关系：

- 上一个完整自然周的数据，只用于生成本周行动建议。
- 本周新增或补录的数据，不补生成本周建议，只用于判断下周是否具备生成条件。
- 上周数据不足时不向前追溯两周或四周，避免把不同阶段的数据混在一份周建议中。
- 系统不自动生成。老师需要在下一周进入班主任助理并选择班级后发起生成。

### 3.3 为什么按上一个完整自然月复盘

“我的评价复盘”使用上一个完整自然月，而不是近4周或自然周：

- 自然月边界清晰，老师能够直接理解“复盘上个月记录”。
- 每个复盘月只生成一份，便于保存和查看历史。
- 当前尚未结束的月份不参与本次生成，避免同一份报告持续变化。
- 上月数据不足时不跨月累计，也不读取更早月份补足。

## 4. 目标

### 4.1 产品目标

1. 让班主任在3分钟内看懂当前最重要的班级信号。
2. 每份报告最多给出3条本周重点行动，避免信息过载。
3. 数据不足时不强行判断学生，也不生成低价值报告。
4. 从产品机制上避免同一批数据和同一班级报告被重复生成。
5. 在免费阶段控制模型文本消耗量，并支持平台规模化使用。
6. 让班主任在3分钟内理解自己的评价视角和证据质量，并获得不超过3条下月记录建议。
7. 让班主任从上周评价明细中理解教师团队的评价方式，而不是重复查看教师评价次数排行榜。

### 4.2 关键结果

| 关键结果 | 目标 |
|---|---:|
| 同一班级同一行动周的模型重复调用率 | 小于1% |
| 数据不足时误调用模型的比例 | 0% |
| 报告行动建议数量 | 不超过3条 |
| 默认展开的报告正文长度 | 500至700个汉字 |
| 含“查看依据”的完整结构化输出 | 不超过1000个汉字 |
| 学生洞察只复述次数、分数或单条评价的报告比例 | 0% |
| 评价洞察只复述教师次数、占比或排名的报告比例 | 0% |
| 班级洞察只复述记录数量或整体好坏的报告比例 | 0% |
| 已生成报告重复访问的缓存命中率 | 不低于95% |
| 有权限用户查看历史报告成功率 | 不低于99% |
| 评价复盘只复述统计数据的报告比例 | 0% |
| 评价复盘下月记录建议数量 | 不超过3条 |

以上目标值为首版验收口径，试点后可以根据真实数据调整。

## 5. 用户与使用范围

### 5.1 核心用户

- 班主任：生成和查看本人带班班级的行动建议。
- 副班主任：在学校授权后查看同一班级的本周行动建议，默认与班主任共享结果。
- 我的评价复盘仅供评价记录本人查看；其他班主任、副班主任和管理员默认不能查看个人复盘正文。

### 5.2 不在首版范围

- 普通任课教师独立生成班级行动建议。
- 自由聊天、连续追问和语音对话。
- 用户手动选择任意日期范围。
- 普通用户重新生成同一行动周报告。
- 实时预警和每条新记录到达后的自动更新。
- 跨班级、跨年级和跨学校综合分析。
- 自动把行动建议加入待办或考核。

## 6. 价值主张

### 6.1 对班主任

- 不需要逐条翻阅全班记录。
- 快速看到“需要留意、值得肯定、应该采取什么行动”。
- 理解不同任课教师如何参与本班评价、采用何种引导方式以及提供了哪些互补观察。
- 能查看往期建议，避免同类问题反复处理却没有复盘依据。
- 数据不足时不生成报告，避免被不可靠结论误导。
- 能按月复盘自己更容易关注谁、记录什么，以及评价表达是否有足够证据。

### 6.2 对学校和平台

- 提高评价记录被实际使用的价值。
- 控制免费功能的模型成本。
- 通过共享报告、缓存和生成去重，避免重复消耗。
- 为后续行动跟进、班级管理复盘和付费能力积累数据基础。

## 7. 解决方案

### 7.1 核心概念

| 名称 | 定义 |
|---|---|
| 行动周 | 建议主要执行的当前自然周，周一00:00至周日23:59 |
| 数据窗口 | 固定为行动周之前的一个完整自然周 |
| 复盘月 | 当前月份之前的一个完整自然月，例如7月生成6月复盘 |
| 数据水位线 | 上一份有效报告已经分析到的最后一条记录编号或时间 |
| 新增数据 | 数据水位线之后、尚未用于正式报告的评价记录 |
| 有效报告 | 已完成AI分析并保存的行动建议或评价复盘 |
| 生成门槛 | 上一个完整自然周的记录数量和学生覆盖率同时达到最低要求 |
| 原始评价事件 | 教师一次提交、语音或批量评价产生的一次评价互动；面向全班时仍计为一个原始事件 |
| 学生明细记录 | 原始评价事件按评价对象展开后写入学生档案的明细；一次全班评价可能生成多条学生明细 |

### 7.2 页面结构

#### 7.2.1 班主任助理首页

入口：教师手机端 → 我的 → 管理工具 → 班主任助理。

首版提供两个方向：

1. 生成班级本周行动建议。
2. 我的评价复盘。

“生成班级本周行动建议”的描述固定为：`根据上周评价，整理本周关注重点和跟进建议。`

“我的评价复盘”的描述固定为：`复盘上月记录，发现评价视角中的盲区与改进方向。`该入口属于阶段性复盘，视觉权重低于本周行动建议，但保持可直接进入。

班级选择规则：

- 只有一个带班班级时，点击后直接生成或查看已有报告。
- 有多个带班班级时，点击后打开“选择班级”底部弹窗。
- 点击班级即进入生成流程，不再增加确认按钮。
- 只展示当前教师拥有班主任权限的班级。
- 默认选中上次生成或查看的班级。

#### 7.2.2 本周行动建议页

页面顶部：

- 左侧返回按钮。
- 当前“本周行动建议”明细页中间不重复展示页面标题；历史详情与报告示例展示各自标题，不展示助理形象。
- 内容区顶部只展示班级名称和“根据{数据周期}评价记录生成”。
- 功能名称“本周行动建议”已经说明建议适用于当前周，不再展示“建议周期”字段。
- 右侧使用历史图标进入“往期建议”。
- 生成等待阶段不展示数据周期、记录数、覆盖人数或参与老师数；报告生成完成后才展示数据来源。

生成等待阶段固定逐行展示：

- 正在整理上周评价记录 → 正在分析评价详情 → 正在归纳班级重点 → 正在生成本周行动建议。
- 不显示百分比、骨架卡片和完成勾选；当前步骤使用轻微呼吸圆点。
- 以上步骤是等待反馈，不代表模型内部的精确执行节点。

报告完成后的数据来源固定为：`根据7月7日-13日评价记录生成`。

数据充足报告最多包含：

1. 学生洞察。
2. 评价洞察。
3. 班级洞察。
4. 本周重点行动，最多3条。

不再单独展示“班级速览”。记录数、覆盖人数和整体提醒信号已经分别存在于统计报表、学生洞察和班级洞察中，继续保留会造成重复。

三类洞察默认只展示对象、模式判断和一条关键证据。解释、证据边界、发生条件和核实重点通过每条洞察下方的“查看依据”渐进展开；展开控件保持次级权重，触控区域不小于44×44像素，支持键盘操作和展开/收起状态。

“评价洞察”分析上周各位教师在本班的评价参与结构和评价方式，不是教师排行榜或教师考核。洞察数量不设置固定上限，只展示能够从统计与评价明细共同验证的有用信息；没有实质信息时隐藏该板块。

页面底部固定展示：`以上内容由AI基于评价记录生成，仅供参考`。

上周数据不足时不进入加载状态，也不调用AI。页面从上到下展示：

1. 原因和结果合并为唯一主信息并逐字展示，固定分为两行：第一行`上周数据不足，`，第二行`暂时无法生成本周行动建议。`
2. 主信息展示完成后，“上周数据”中的记录条数、覆盖同学两项依次出现；每项显示实际上周值、动态目标值以及“还差{差额}”或“已达标”。“上周数据”是页面唯一主要卡片。
3. 上周目标按上周数据周期结束时的有效在班人数计算；当前周目标按当前有效在班人数计算，班级人数变化时分别使用对应周期目标。
4. 本周未达到条件：`本周达到以上条件后，下周进入班主任助理即可生成。`
5. 本周已达到条件：`本周已达到以上条件，下周进入班主任助理即可生成。`
6. 两项上周数据展示完成后，再出现下周生成说明和`查看报告示例`入口。
7. `查看报告示例`保持最低权重，使用普通文字与向右箭头；示例必须明确标记，不写入往期建议。
8. 数据不足页不提供`记录本周表现`快捷入口，避免低频建议页侵入老师日常记录流程。
9. 系统开启减少动态效果时，主信息和两项数据直接完整展示，不强制等待动画。

数据不足页不使用“再记录几位即可生成本周建议”“下周将为你生成”等文案，避免暗示本周可以补生成或系统会自动生成。

#### 7.2.3 往期建议页

- 页面名称：“往期建议”。
- 按建议开始日期归入对应年月，年月分组和组内建议均倒序排列。
- 月份使用“2026年7月”形式，左侧使用细竖线和小圆点建立时间层级，不增加状态或操作。
- 页面提供轻量班级切换；只有一个带班班级时显示普通班级名称，多个班级时点击名称打开共用的“选择班级”底部弹窗。
- 切换班级只改变当前历史浏览范围，不要求返回班主任助理重新进入生成流程。
- 列表只展示成功生成的报告，数据不足和生成失败不进入历史列表。
- 每条只展示精确的建议日期范围和“基于{数据周期}评价记录”；页面已表明为往期建议，列表不再重复“行动建议”。
- 不使用“第几周”命名建议，避免月初起算日和跨月周期的口径歧义。
- 不展示排序说明、摘要、记录数、覆盖人数和状态标签。
- 点击任一记录进入只读报告详情。
- 历史详情不再显示历史入口，避免页面循环跳转。

#### 7.2.4 我的评价复盘页

生成流程：

1. 点击“我的评价复盘”，一个带班班级时直接进入；多个带班班级时复用“选择班级”底部弹窗。
2. 系统只检查班主任本人在该班级上一个完整自然月的评价记录。
3. 达到记录量门槛后调用AI；当前复盘月已经生成时直接读取缓存。
4. 生成等待固定逐行展示：`正在整理上月评价记录 → 正在分析关注对象 → 正在核对指标与表达 → 正在生成评价复盘`。

报告页不展示独立统计总览。页面从上到下展示：

1. 本月评价画像：概括记录呈现出的观察模式，不复述数量和占比。
2. 关注对象：解释哪些学生更容易或更难进入记录视野，以及通常由什么事件触发。
3. 评价视角：分析教师更容易记录纠偏、明显成果、集体活动还是日常过程。
4. 指标与表达：分析行为与指标是否匹配，是否存在归因过早、标签化或情境不足等评价表达质量问题。
5. 下月记录建议：最多3条低成本、可执行的观察和记录改进动作。

页面右上角使用历史图标进入“往期复盘”。历史按自然月倒序排列，支持切换带班班级；点击月份进入只读详情。每个班级每个复盘月只生成一份，历史报告不随之后的记录修改自动变化。

数据不足时不调用AI，页面只展示`上月记录不足，暂时无法生成评价复盘。`、实际上月记录量、动态目标量和报告示例。数据不足结果不写入历史。

### 7.3 数据窗口规则

#### 7.3.1 当前行动周

1. 系统只检查行动周之前的一个完整自然周。
2. 上周达到门槛时，老师进入班主任助理并选择班级后发起生成。
3. 上周未达到门槛时，不调用AI、不生成报告，也不写入往期建议。
4. 不向前读取两周、四周或更早记录。
5. 不使用当前尚未结束的自然周补生成本周报告。

#### 7.3.2 为下周准备

1. 数据不足页单独统计本周当前记录量和学生覆盖人数。
2. 本周补录允许计入本周统计，不考核记录分布天数。
3. 本周达到门槛，只表示下周具备生成条件，不触发自动生成。
4. 下周老师再次进入班主任助理并选择班级后，系统基于本周完整数据发起生成。
5. 本周未达到门槛时，下周仍展示数据不足页，不跨周累计。

#### 7.3.3 同一周内的数据变化

- 上周数据是否充足在本行动周开始时确定，本周补录不改变本周报告状态。
- 本周一旦生成成功，本周后续新增数据不更新本周报告，只作为下周数据来源。
- 已生成报告重复访问直接读取缓存，不提供普通用户重新生成入口。

#### 7.3.4 评价复盘月度窗口

1. 只读取当前月份之前的一个完整自然月。例如2026年7月进入时，只分析2026年6月1日至30日。
2. 只读取当前教师本人、当前选择班级的记录，不混入其他教师记录。
3. 当前尚未结束的月份不参与本次报告，也不能补生成上个月复盘。
4. 上月记录不足时不读取更早月份，不跨月累计。
5. 当前月记录只用于下个月复盘；每月进入后仍由老师主动发起，系统不自动生成。
6. 已生成复盘重复访问读取缓存，后续修改历史记录不自动更新报告。

### 7.4 数据充足度

首版建议使用可配置门槛，不把数值写死在前端。

生成条件必须同时满足：

| 条件 | 首版规则 |
|---|---:|
| 记录数量 | 不少于班级人数的60%，结果向上取整 |
| 学生覆盖率 | 不少于班级人数的50%，结果向上取整 |

例如20人班级需要不少于12条记录并覆盖10位同学，45人班级需要不少于27条记录并覆盖23位同学，60人班级需要不少于36条记录并覆盖30位同学。门槛由服务端配置并返回换算后的目标条数和目标人数，前端只展示具体数字，不展示百分比规则。

允许老师集中补录，不检查活跃记录天数。不能只依赖记录条数：大量记录只覆盖少数学生时，仍判断为数据不足。学生洞察不设置独立的报告生成门槛；证据较少时通过谨慎表达和核实建议控制风险，不阻止整份报告生成。

我的评价复盘使用独立门槛：上一个完整自然月的记录数量不少于`班级人数的50%`，向上取整，且最低不少于15条。例如20人班目标15条、45人班目标23条、60人班目标30条。

评价复盘不把学生覆盖率作为生成门槛。学生是否集中在少数对象上，本身就是报告需要分析的内容；如果用覆盖率阻止生成，反而无法发现这一记录盲区。活跃记录天数和正负向比例也不作为生成门槛。

### 7.5 内容生成规则

#### 7.5.1 输入

优先复用平台已有的结构化评价解析结果：

- 原始事件编号。
- 评价对象编号、姓名以及对象范围，明确区分个人、小组和全班。
- 评价人、任教学科。
- 记录时间和事件时间。
- 教师原始评价文本或必要转写、结构化行为摘要。
- 匹配指标。
- 正向或负向。
- 分值。
- 原始评价事件展开后的学生明细记录数。
- 如已有结构化数据，可附带学科、课堂或任务阶段、任务难度、完成质量、行为发生前情境和后续反应。

本周行动建议必须同时提供后端预先计算的教师评价结构化统计，至少包括：

- 班级原始评价事件总数、展开后的学生明细记录总数。
- 每位教师的原始评价事件数及占比、学生明细记录数、覆盖学生数。
- 每位教师的正向、负向和中性事件数。
- 每位教师面向个人、小组和全班的原始评价事件数。
- 每位教师按场景的事件分布和后续跟进事件数。

精确计数与占比由后端完成，AI不得从长文本明细重新计算。评价参与度统一使用原始评价事件数；学生明细记录数只用于解释全班或多人评价展开后的影响，两个口径不得混用。

同时提供学生和班级模式结构化统计：

- 学生统计区分明确面向个人的原始事件、参与小组事件和因全班评价展开产生的明细，并提供评价教师数、场景分布、正负向分布、跟进事件数和按时间排序的事件编号。
- 班级模式统计提供共性候选对应的独立事件数、不同学生数、不同教师数、学科与任务阶段、个人/小组/全班事件构成和代表事件编号。
- 结构化统计只用于定位候选模式；AI必须回到对应原始事件核实行为、场景和时间顺序后才能输出洞察。

可选情境字段缺失时传空或省略，不使用模型补全。单条模糊评价不足以支撑原因洞察时，只能给出核实方向。

不重复发送：

- 完整语音文件。
- 冗长语音转写原文。
- 已有统计图表。
- 与本次分析无关的上学期或更早记录。
- 同一原始事件产生的重复文本。

#### 7.5.2 输出

本周行动建议AI必须返回固定结构的JSON，不直接返回不可解析的长文章。同一类型的所有成功报告字段完全一致，没有内容时返回空数组：

```json
{
  "student_insights": [
    {
      "student_names": ["涉及的学生姓名"],
      "insight_type": "continuity | context_contrast | change | cross_teacher | strength | significant_event",
      "finding": "跨记录形成的学生模式判断",
      "evidence": "独立事件、教师和场景证据",
      "interpretation": "可能解释、育人意义或证据边界",
      "verification_focus": "需要进一步确认的关键信息；无需核实时为空字符串",
      "need_verify": true
    }
  ],
  "evaluation_insights": [
    {
      "teacher_names": ["涉及的教师姓名，可为多人"],
      "insight_type": "participation | orientation | target_scope | content_quality | follow_up | complementarity",
      "finding": "教师评价模式判断",
      "evidence": "后端精确统计和代表性明细特征",
      "implication": "对班级评价观察的意义或边界"
    }
  ],
  "class_insights": [
    {
      "insight_type": "shared_pattern | context_pattern | change | strength | differentiation",
      "finding": "班级模式判断",
      "evidence": "涉及学生、教师、独立事件和场景范围",
      "condition": "模式集中发生的条件；无明显条件时为空字符串",
      "implication": "对本周班级行动的意义或证据边界"
    }
  ],
  "actions": []
}
```

内容要求：

- 只描述观察到的行为，不给学生贴标签。
- 使用学生投入三维框架、成就情绪“控制-价值”理论、自我决定理论、挑战-能力匹配和功能性行为评估作为备选解释框架，不得用于个体诊断。
- 持续或重要的负向信号应形成最多2种可区分的解释，同时写明证据边界；信息不足时不补全动机。
- 五育指标是记录分类，不是原因诊断；命中德育指标不等于“德育出了问题”。
- 学生洞察优先识别连续信号、情境反差、变化轨迹、跨教师印证或差异、跨场景优势和高价值单次事件，不按正负向凑数。
- 每条学生洞察必须包含学生对象、跨记录模式、独立事件证据以及解释或证据边界；次数、分数、排名和单条普通表扬不构成洞察。
- 全班评价展开后的学生明细不能代替个体观察。`class` 事件只能作为共同情境证据，不能单独支撑个体模式。
- 学生洞察不再承载具体执行动作。需要核实时只填写 `verification_focus`，本周执行动作统一收敛到 `actions`，避免重复。
- 评价洞察优先识别评价参与结构、鼓励与纠偏方式、个人/小组/全班对象粒度、观察场景、表达质量、持续跟进和教师视角互补。
- 每条评价洞察必须包含教师对象、模式判断、明细证据以及对班级评价观察的意义或边界；单独的次数、占比、覆盖人数或排名不构成洞察。
- 评价洞察不设置固定条数上限，也不要求覆盖每位教师；同类结论合并，没有有用信息时返回空数组。
- 只描述“本周评价呈现的倾向”，不得推断教师长期育人理念、能力、态度、责任心或教学质量。
- 正向评价多、负向评价多或全班评价多均不能单独作为好坏判断，必须结合原始内容、评价对象、场景和后续记录解释。
- 班级洞察优先识别跨学生共性、情境模式、班级变化、优势机制和分化信号。
- 每条班级洞察必须包含班级模式、涉及范围、发生条件和行动意义；某类记录较多、整体表现较好或单次全班评价不构成稳定班级模式。
- 单个学生的重复行为不得上升为班级共性；一次全班评价展开出的多条学生明细仍只算一个独立事件。
- 数据中等时减少结论数量，明确需要核实。
- 记录少不等于学生表现少或存在问题。
- 推测性判断必须带“建议核实”。
- 行动必须具体，避免“加强教育”等空话。
- 学生姓名、班级和记录必须来自授权数据。

研究框架完整规则、归因边界和论文来源见《班主任助理_本周行动建议_内容生成提示词》，框架版本为 `educational_insight_framework_v1`。实际调用不传入论文全文或摘要。

#### 7.5.3 我的评价复盘输入与输出

正式调用必须同时提供：

- 上一个完整自然月内，当前教师对当前班级提交的全部评价记录明细。
- 当前复盘月内的完整班级学生名单；没有名单时不得判断哪些学生未进入记录视野。
- 学校当前生效的完整指标体系，包括层级、定义、行为示例和计分规则。
- 后端预先计算的结构化统计，包括按学生、正负向、指标和场景的分布。

精确计数由后端完成，AI不负责从长文本中重新计算条数或比例。AI使用结构化统计定位模式，再回到原始评价明细核实和解释。输出不得只复述数量、占比或排名，必须提供以下解释性洞察：

1. 关注对象：高频和低频进入记录的学生群体、常见触发事件以及证据边界。
2. 评价视角：教师更容易在纠偏、明显成果、集体活动或日常过程中开始记录，哪些过程性证据较少进入记录。
3. 指标使用：行为与指标是否匹配，是否过度集中在少数宽泛指标；不要求机械覆盖全部指标。
4. 评价表达质量：是否区分观察事实与评价判断，是否存在归因过早、标签化、情境不足或集体描述过泛。
5. 下月行动：最多3条改善观察方式和证据质量的具体动作。

AI必须返回固定JSON：

```json
{
  "review_overview": [],
  "attention_insights": [],
  "perspective_insights": [],
  "indicator_and_expression_insights": [],
  "actions": []
}
```

完整规则见《班主任助理_我的评价复盘_内容生成提示词》。

### 7.6 生成去重规则

#### 7.6.1 报告唯一键

服务端以以下字段确定同一份报告：

```text
学校编号 + 班级编号 + 行动周开始日期
```

同一唯一键最多存在一份成功报告。

我的评价复盘使用独立唯一键：

```text
学校编号 + 教师编号 + 班级编号 + 复盘月份
```

每个班级每个复盘月只生成一份个人复盘。同一班级的不同班主任或副班主任分别生成自己的报告，不共享正文。

#### 7.6.2 生成任务状态

以下状态只供后端控制生成任务，不属于报告JSON，也不在往期建议或往期复盘中展示：

| 状态 | 含义 | 用户看到的结果 |
|---|---|---|
| not_started | 未检查、未生成 | 可以发起检查 |
| insufficient_data | 上一个完整自然周未达到门槛 | 展示上周不足、本周进度和报告示例 |
| queued | 已进入任务队列 | 等待生成 |
| generating | 模型正在生成 | 共享同一加载任务 |
| generated | 已生成 | 直接读取缓存报告 |
| failed | 生成失败 | 展示可操作错误和系统重试状态 |

#### 7.6.3 并发处理

- 多位有权限老师同时点击，只创建一个生成任务。
- 其他请求复用同一个任务编号并等待结果。
- 生成成功后所有人读取同一份报告。
- 生成失败允许系统自动重试，建议最多2次。
- 普通用户不能通过重复点击创建新任务。

#### 7.6.4 重新生成

首版不向普通用户提供“重新生成”。

以下情况可以由系统或管理员重新处理：

- 报告生成失败且自动重试仍失败。
- 数据错误被管理员修复。
- 模型输出无法解析或触发安全拦截。

重新处理仍复用原报告唯一键，并记录版本，不能额外创建同一周期的第二份报告。

### 7.7 数据水位线规则

每份成功报告至少保存：

- 本次使用的最早记录时间。
- 本次使用的最晚记录时间。
- 最大记录编号。
- 记录数量。
- 固定结构的报告内容。

数据水位线用于保存本次报告的数据快照和审计依据，不用于跨周期累计。本周行动建议下一次仍只读取新的上一个完整自然周；评价复盘下一次仍只读取新的上一个完整自然月。

删除或修改历史记录时：

- 普通修改不自动重生成已发布报告。
- 修改后的记录进入下一周期分析。
- 重大数据错误由管理员触发修复流程。

### 7.8 模型成本约束

#### 7.8.1 调用约束

- 每班每行动周最多成功调用一次。
- 数据不足时不调用模型。
- 无人查看的班级不提前生成，首版采用首次访问时生成。
- 已生成报告重复访问全部读缓存。
- 每位教师、每个班级、每个复盘月最多成功调用一次评价复盘模型。

#### 7.8.2 输入压缩

- 对同一原始事件去重。
- 使用短字段，不发送重复说明。
- 本周行动建议超过配置上限时保留完整统计，并按学生、正负向、指标和日期分层选取代表记录。
- 教师评价结构化统计始终保留完整教师口径；代表记录还应按评价人和个人/小组/全班对象范围分层选取，避免抽样改变教师评价画像。
- 学生和班级模式统计始终保留完整口径；代表记录还应覆盖候选模式对应的不同学生、教师、场景和时间顺序，避免抽样抹掉反差或变化。
- 本周行动建议默认原始证据记录上限建议为120条，后续按模型和成本评估调整。
- 不传上一份报告或更早记录，避免模型跨周延伸结论。
- 评价复盘必须让上月全部明细参与分析，不随机抽样。超过单次上下文时，按完整记录分批提取证据后再合并，不得只保留高分、负向或少数学生记录。
- 评价复盘同时传入后端结构化统计，避免模型承担精确计数。

#### 7.8.3 输出限制

- 默认展开的报告正文控制在500至700个汉字，完整结构化输出不超过1000个汉字。
- 学生洞察建议不超过3项。
- 本周重点行动最多3条。
- 避免重复复述记录数量和统计图表，不设置班级速览字段。
- 评价洞察不限制固定条数，但必须服从全文长度上限，只保留无法从统计报表直接得到的解释性结论并合并同类项。
- 评价复盘输出最多1000个汉字，解释性洞察每类最多2条，下月记录建议最多3条。

#### 7.8.4 平台保护

- 设置平台每日调用预算。
- 超过预算后进入排队或延迟生成，不重复提交任务。
- 监控单份报告输入量、输出量、生成时长、缓存命中率和失败率。

### 7.9 权限与隐私

- 用户只能查看本人有带班权限的班级报告。
- 班主任和已授权副班主任共享同一班级报告。
- 上一条仅适用于本周行动建议。我的评价复盘只允许记录本人查看，不与其他教师共享。
- 班级关系变化后立即重新校验查看权限。
- 报告包含未成年人信息，不允许公开分享链接。
- 服务端记录生成者、查看者、班级、时间和报告版本。
- 页面不展示模型内部提示词、成本、数据水位线等实现信息。

本周行动建议接口中的 `record_count` 统一表示原始评价事件数，`student_entry_count` 表示按学生展开后的明细记录数。前端和AI服务不得将两个字段混用。

生成任务状态通过独立接口返回，不混入报告内容。同一报告类型的所有成功报告使用相同的 `content` 字段。

我的评价复盘接口建议至少返回：

```text
review_id
school_id
teacher_id
class_id
class_name
review_month
availability_status
data_period_start
data_period_end
record_count
target_record_count
generated_at
data_watermark
prompt_version
content_version
content.review_overview
content.attention_insights
content.perspective_insights
content.indicator_and_expression_insights
content.actions
```

### 7.11 异常与边界场景

| 场景 | 处理方式 |
|---|---|
| 班主任没有带班班级 | 不展示生成入口，提示先完善带班信息 |
| 班主任有多个带班班级 | 进入前选择班级，记住上次选择 |
| 数据不足 | 不调用AI、不创建报告，展示上周不足、本周进度和报告示例 |
| 正在生成 | 展示同一任务进度，禁止重复提交 |
| 网络中断 | 允许退出；再次进入读取任务状态 |
| 生成失败 | 展示稍后查看，不要求用户连续点击 |
| 输出解析失败 | 服务端重试或进入失败状态，不展示原始模型文本 |
| 权限被移除 | 立即禁止查看当前和历史报告 |
| 本周已经生成 | 直接打开报告，不显示重新生成 |
| 本周生成后新增记录 | 新记录进入下一次生成候选 |
| 上月评价复盘记录不足 | 不调用AI、不创建历史，展示记录量和报告示例 |
| 上月评价复盘已生成 | 直接读取个人缓存报告，不提供重新生成 |
| 缺少完整班级名单 | 不输出未被关注学生结论，服务端记录数据缺失 |
| 学校指标体系缺少定义 | 不分析指标错配，只保留有证据的关注对象和表达洞察 |

### 7.12 Demo mock要求

当前汇报 Demo 固定展示：

1. 2025级四班：上周数据充分，展示完整报告和生成过程。
2. 2025级一班：上周数据不足，本周尚无记录。
3. 2024级二班：上周数据不足，本周正在积累记录。
4. 2025级七班：上周数据不足，本周记录已达到下周生成条件。
5. 数据不足的三个班级均可查看完整报告示例；示例不进入往期建议。
6. 多个班级提供成功生成的往期建议，用于展示历史班级切换和空历史状态。
7. 2025级四班、2025级七班：上月记录达到评价复盘门槛，展示有意义的AI复盘和往期月度历史。
8. 2025级一班、2024级二班：上月记录不足，展示不同记录量和评价复盘示例。
9. 完整周报至少展示评价参与结构、鼓励与纠偏方式、评价对象粒度中的多类教师评价洞察，且不能只展示教师次数和排名。
10. 学生洞察至少展示跨场景优势、情境反差或变化轨迹中的多类信号，不能只展示单条表扬和提醒。
11. 班级洞察展示有涉及范围和发生条件的共性信号或优势机制，不能只展示“记录较多”“整体较好”。

Demo 中不展示数据等级切换器，所有状态由班级 mock 数据驱动。同类历史报告使用相同JSON字段，不展示数据等级、摘要或状态标签。

### 7.13 验收标准

#### 页面与交互

- 本周行动建议右上角存在可点击的历史图标，触控区域不小于44×44像素。
- 图标具备“查看往期建议”的无障碍名称。
- 往期建议按执行周期倒序排列。
- 多班级教师点击生成入口时先选择班级；单班级教师不出现额外选择步骤。
- 往期建议可以在页面内切换带班班级，生成入口与历史页复用同一班级选择组件。
- 历史列表可进入由对应上一个完整自然周生成的报告。
- 历史详情返回后回到历史列表，再返回回到当前报告。
- 历史详情不重复显示历史入口。
- 当前报告顶部不展示助理形象和“建议周期”字段。
- 加载阶段不展示数据来源，报告完成后才显示。
- 加载阶段不展示记录数、覆盖人数、参与老师数和骨架卡片。
- 生成等待只展示固定的上周分析步骤，不出现两周、四周追溯文案。
- 数据不足页先逐字展示原因，再逐条展示上周记录结果，最后展示下周说明和报告示例；不展示记录入口。
- 本周达到条件时明确提示下周需进入班主任助理发起生成，不暗示自动生成。
- 历史卡片不展示摘要、统计和状态标签。
- 当前完整报告正文明显短于旧版，行动建议不超过3条。
- 当前报告不展示班级速览；三类洞察默认只展示简洁结论和关键证据，完整解释通过“查看依据”展开。
- “查看依据”支持展开和收起，键盘可操作，触控区域不小于44×44像素。
- 评价洞察数量不设置固定上限，只展示由统计与明细共同支持的有用信息；同类结论合并。
- 学生洞察不超过3条，每条展示跨记录模式和独立事件证据，不在洞察中重复本周重点行动。
- 班级洞察不超过2条，每条展示涉及范围、发生条件和行动意义。
- “我的评价复盘”入口名称和描述明确使用上一个自然月，视觉权重低于本周行动建议。
- 评价复盘生成等待依次展示整理记录、分析关注对象、核对指标与表达、生成复盘。
- 评价复盘正文不设置统计总览，固定展示本月评价画像、关注对象、评价视角、指标与表达、下月记录建议。
- 评价复盘右上角可进入往期复盘，历史按自然月倒序并支持班级切换。
- 评价复盘数据不足时不展示AI结论，允许查看报告示例。

#### 业务规则

- 同一学校、班级、行动周并发请求只创建一个任务。
- 数据不足时不调用模型。
- 已生成时重复访问只读取缓存。
- 本周生成后新增记录不触发本周重新生成。
- 每份报告只读取行动周之前的一个完整自然周。
- 本周补录不改变本周报告状态，也不补生成本周报告。
- 本周达到门槛不自动生成，下周仍由老师进入班主任助理发起。
- 老师只能选择班级，不能手动选择分析时间范围。
- 评价复盘只读取当前教师、当前班级上一个完整自然月的数据。
- 评价复盘记录量达到动态目标即可生成，不把学生覆盖率作为生成门槛。
- 每个班级每个复盘月只生成一份，重复进入读取缓存。
- 精确统计由后端计算，AI使用全部明细和指标体系解释记录模式。
- 本周行动建议的教师参与度按原始评价事件统计，不使用全班评价展开后的学生明细数代替。
- 学生个体模式优先使用个人或可明确识别学生行为的小组事件，全班评价不能单独支撑个体结论。
- 班级共性必须由多个学生和多个独立事件共同支持，单个学生或一次全班评价不能上升为稳定班级模式。

#### 内容安全

- 推测性结论提醒老师核实。
- 持续或重要的负向信号最多提出2种可区分解释，并明确现有证据边界。
- 对“上课不认真听讲”等行为不直接归因为调皮、态度或德育问题。
- 每条学生原因假设都说明证据边界和核实重点；具体执行动作统一放入本周重点行动，记录不足时不生成原因结论。
- 不输出家庭教育、人格、智力、医学、心理或神经发育诊断。
- 研究框架使用 `educational_insight_framework_v1`，不向模型传入论文全文或摘要。
- 所有生成报告显示“以上内容由AI基于评价记录生成，仅供参考”。
- 评价洞察不得只复述教师次数、占比、覆盖人数或排名，必须结合评价对象、原始内容、场景或后续记录形成模式判断。
- 评价洞察不得根据一周记录推断教师长期育人理念、能力、态度、责任心或教学质量，不得把全班评价直接判断为低质量。
- 评价复盘不得把记录分布直接解释为教师偏见、学生真实表现分布或教师能力结论。
- 评价复盘不得只复述数量、占比或排名，每条洞察必须回到评价明细说明证据边界。

## 8. 发布计划

### 阶段一：领导汇报Demo

- 当前数据充分报告。
- 精简后的报告内容。
- 往期建议入口、列表和详情。
- 数据不足的未记录、积累中、已达标三种 mock。
- 数据不足页的完整报告示例。
- 我的评价复盘完整报告、数据不足状态、报告示例和往期月度历史。
- 本 PRD 及生成提示词文档。

### 阶段二：正式数据接入

- 班级权限和班级选择。
- 数据充足度计算。
- 固定上一个完整自然周的数据窗口。
- 生成任务、缓存、唯一键和并发控制。
- 数据水位线。
- 历史报告存储和查询。
- 真实AI结构化输出。
- 上月全部评价明细、班级名单、学校指标体系和后端结构化统计的评价复盘调用链路。
- 个人评价复盘的权限隔离、月度唯一键和历史查询。

### 阶段三：体验与运营优化

- 上次报告延续判断。
- 生成排队和成本看板。
- 学生原始记录核实入口。
- 行动建议完成状态和后续复盘。
- 根据试点数据调整生成门槛。
- 跨月趋势对比和上月建议执行后的变化复盘。
